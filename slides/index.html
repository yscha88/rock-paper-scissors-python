Code를 이해한다는 것

프롬프트 엔지니어링 시대에도 코딩을 배우는 이유 — 이 코드에는 버그가 하나 있다. 코드를 모르면 이 버그를 못 찾는다.

life.c — 인생 1년
for (int i=0; i<365; i++) {   // 하루 단위로 1년
    money++;                  // 매일 돈을 번다 → OK
    age++;                    // ???
    if (age > 30)
        changeJOB(PhD);
}

age++ 가 하루 루프 안에 있다. 하루에 한 살씩 → 1년에 365살. 한 달이면 30을 넘겨 changeJOB이 엉뚱한 시점에 발동한다. 그리고 수정은 한 줄 이동으로 끝나지 않는다:

for (int year=0; year<80; year++) { // ① 바깥 '해' 루프가 더 필요
    for (int day=0; day<365; day++)
        money++;               //    매일 버는 건 안쪽
    age++;                     // ② 한 해가 끝나야 한 살
    if (age > 30)              // ③ 검사는 나이가 "변한 직후"
        changeJOB(PhD);        //    — 매일 검사는 낭비+오류 여지
}

논리 버그의 수정은 종종 구조(루프 중첩)를 다시 세우는 일이다 — 코드를 읽을 줄 알아야 여기까지 보인다. 그것이 이 저장소의 존재 이유다.

레시피와 요리사 — 기본 용어부터

소스코드는 "한국어로 쓴 요리 레시피", CPU는 한국어를 못 읽고 기계어(0/1)만 아는 요리사다.

단어쉽게 말하면
프로그램컴퓨터에게 시키는 "할 일 목록" — 가위바위보도 프로그램

실행(run)코드에 적힌 대로 컴퓨터가 실제로 일하게 하는 것
📝 소스코드사람이 읽는 글
── 번역 ──▶
🔢 기계어0과 1
──▶

AI 시대에는 층이 하나 더 얹혔다

번역의 층
# 예전
소스코드 ──[ 번역 ]──▶ 기계어(0/1) ──▶ CPU 실행

# 요즘
프롬프트(사람의 말) ──[ AI 가 코딩 ]──▶ 소스코드
        ──[ 번역 ]──▶ 기계어(0/1) ──▶ CPU/GPU 실행
  • 번역 층이 하나 늘었을 뿐 — 각 층의 번역은 여전히 틀릴 수 있다.
  • AI 가 만든 코드에 age++ 같은 논리 버그가 섞여도 기계는 군말 없이 실행한다.
  • 중간층(소스코드)을 읽을 줄 아는 사람만이 그 번역을 검증할 수 있다 — 3번 슬라이드의 논지.
  • AI 학습·추론 같은 대규모 병렬 계산은 GPU(수천 개의 단순 계산기를 묶은 칩) 담당.

01 · CS 기초

"컴파일"의 속살 — 어셈블리라는 중간역

C/C++ 번역 파이프라인
소스코드(.cpp)
 → [전처리기]  #include 펼침
 → [컴파일러]  어셈블리(.s)   ← 사람이 읽는 기계어 표기
 → [어셈블러]  목적 파일(.o)  ← 진짜 0/1 조각
 → [링커]      실행 파일       ← 주소 연결·이어붙임

링커가 연결할 함수를 못 찾으면 → 그 유명한 undefined reference.

한 줄이 어떻게 바뀌나 (x86-64, 개념)
// 원래 코드 — 판정식 한 줄
int diff = (a - b + 3) % 3;

; ↓ 컴파일러가 만든 어셈블리 (명령 7개)
mov  eax, [a]      ; a 를 레지스터에 담고
sub  eax, [b]      ;   - b
add  eax, 3        ;   + 3
mov  ecx, 3        ; % 3 을 위한 준비 —
cdq                ;   나눗셈 전 부호 확장
idiv ecx           ;   나누면 나머지가 edx 에
mov  [diff], edx   ; diff = 그 나머지

사람의 한 줄 = 기계의 낱개 명령 7개. CPU는 이걸 인출→해석→실행으로 초당 수십억 번 반복 — JVM·파이썬 인터프리터도 같은 구조의 "소프트웨어 CPU"다.

세 언어의 실행 경로

  • Python — 고치고 바로 실행. 작성→실행 사이클 최단, 실행 자체는 최저속.
  • 이 "빌드 없음"이 Python 생산성의 큰 부분.

메모리 — 다 쓴 그릇은 누가 치우나

메모리 = 값을 올려두는 작업대. 빌리고(할당) 반납(해제)해야 하며, 반납을 잊으면 누수.

언어누가 치우나개발자 부담컴퓨터(런타임) 부담
C직접 — malloc/free높음 — 누수·이중 해제 위험낮음 — 정확히 관리하면 최소
C++범위 벗어나면 자동 — RAII·unique_ptr중간낮음
JavaGC(청소부)가 알아서 회수낮음높음 — GC 실행 비용 + 여유 메모리 필요
Python참조 카운팅 + GC낮음중간 — 참조 카운팅은 즉시, 순환 참조만 GC
  • 부담은 사라지지 않는다 — 이동할 뿐. GC는 "개발자의 설거지"를 "기계의 설거지"로 옮긴 것.
  • 그래서 GC 언어는 같은 일에 메모리를 더 쓰는 경향이 있고, GC가 빨리 안 비워 주면 낭비가 쌓이며, 회수가 돌 때 일시 멈춤(pause)도 생긴다.
  • GC가 있어도 관리는 남는다 — 규모가 커지면 GC 선택(JDK별 Parallel→G1→ZGC)·힙 튜닝·참조 수명 관리로 대상이 바뀔 뿐. JVM도 누수 난다(static 컬렉션·ThreadLocal·힙 밖 네이티브 버퍼).
  • 이 낭비가 한계에 닿은 극단이 다음 장의 OOM이다.
01 · 메모리와 OOM

OOM — "메모리 부족"이 전부가 아니다

이름부터 함정이다 — "Out of Memory"라고 해서 램이 정말 모자랐다는 뜻이 아니다. 메시지만으로는 알 수 없고, 실제 현장의 OOM은 대부분 다른 이유다.

① 누수 — "못" 치우는 것

참조를 계속 쥐고 있으면 GC는 "아직 쓰는 중"으로 판단해 치울 수 없다 (쌓기만 하는 전역 캐시가 전형). 메모리가 부족해서가 아니라 회수가 불가능해서 죽는다.

② 한도(limit) 도달

JVM -Xmx, 컨테이너(cgroup) memory limit — 서버 램이 남아돌아도 설정한 상한을 넘으면 OOM. "램은 널널한데 왜 죽지?"의 정체.

③ OOM Killer — 죽임 "당하는" 것

리눅스 커널이 메모리가 몰리면 프로세스를 골라 강제 종료한다. 쿠버네티스의 OOMKilled(exit 137)가 이것 — 앱 내부 오류가 아니라 바깥에서 죽인 것.

진단 순서

OOM을 보면 "램 늘리기" 전에 — 무엇이 안 치워지나?(누수) → 한도가 얼마인가?(-Xmx·limit) → 누가 죽였나?(exit 137) 순으로 본다.

실전 예 — VS Code 창의 reason: 'oom' 크래시: 램 부족이 아니라 그 창(렌더러)의 한도·누수 문제라, 해결책도 램 증설이 아닌 확장 비활성화·캐시 삭제·한도 조정. 그리고 OOM(할당 실패, exit 137) ≠ nullptr 크래시(접근 위반, exit 139) — 단 malloc은 부족하면 NULL을 주므로 OOM이 nullptr 크래시로 위장되곤 한다.

01 · 에러 메시지의 정체

같은 에러에 해결책이 수십 개인 이유

VS Code의 'oom' 크래시는 확장 누수·대용량 파일, 심지어 코딩 폰트+합자(CommitMono·PragmataPro Liga — #239541 확인) 조합으로도 난다. 왜 전부 "oom" 한 단어일까?

에러명은 "사인(死因)"이지 "범인"이 아니다
  • 수많은 원인(누수·폭주 루프·폰트 렌더링 버그)이 전부 "메모리 고갈"이라는 같은 종착지로 흘러든다.
  • 부모 프로세스는 죽은 자식의 exit code만 보고 라벨을 붙인다 — 'oom'은 렌더러 종료 코드의 분류명일 뿐.
  • 같은 메시지 아래 서로 다른 원인을 고친 해결책 수십 개가 모인다 — 공식 이슈(#226484)는 8가지를 시도하고도 원인 미상으로 닫혔다.
  • 라벨→범인 예측은 불가, 역추적만 가능: 재현 조건 격리 → A/B(폰트·합자 끄기) → 덤프 스택 → 프로파일.
  • 실측 — 라벨은 'oom'이어도 이벤트 뷰어엔 "WebView2 보안 정책 위반으로 강제 종료 → 재시작 반복"이 적혀 있기도 하다. 진짜 사인은 OS의 장부에 있다.
예외 처리 계층이 라벨을 바꾼다
같은 할당 실패, 다른 라벨
// C++ — 처리 방식에 따라 다른 죽음으로 보인다
new T;           // bad_alloc 예외 → 안 잡으면 abort
new(nothrow) T;  // nullptr 반환 → 미검사면 segfault!

// Java — OOM Error를 catch로 삼키면?
try { ... } catch (OutOfMemoryError e) { }
// → 반쯤 초기화된 채 살아남아
//   나중에 엉뚱한 NPE로 재등장

라벨은 마지막으로 잡은 계층이 붙인다. 진짜 질문은 "이 라벨, 누가 붙였나?" — 그리고 "fixed"는 사슬 하나가 끊겼다는 뜻일 뿐, 같은 라벨은 다른 범인으로 재발한다.

01 · 실행 모델과 버그

순서가 어긋나면 — nullptr · undefined · NameError

"정의(초기화)보다 사용이 먼저" — 언어마다 증상 이름만 다른 한 가족의 버그.

C++ · 컴파일/링크 시점

파일 간 전역 객체 초기화 순서는 보장 안 됨(static initialization order fiasco) → 초기화 전 참조 = nullptr·쓰레기 값 크래시. 링크 순서가 틀리면 undefined reference.

Python · 실행 시점

위에서 아래로 실행하며 이름을 만듦 — 정의 전에 호출하면 NameError. 순환 import는 "반쯤 초기화된 모듈" → AttributeError.

JavaScript

var 선언은 끌어올려지고(호이스팅) 값 대입은 제자리 — 그 사이에 읽으면 undefined.

세 증상의 뿌리는 하나 — "이 이름(주소)이 쓰이는 시점에 준비돼 있는가?" 실행 모델을 알면 어느 단계(컴파일·링크·실행)에서 터졌는지로 원인을 좁힐 수 있다.

01 · 왜 배우나

01 · Python 특성

강점과 약점

강점 — 사람의 시간을 아낀다
  • 빌드 없음 · 짧은 실행 사이클
  • 적은 코드량 — 동적 타입 + 풍부한 표준 라이브러리
  • 배터리 포함 — secrets·enum·dataclasses 기본 제공
  • 유니코드 기본 — CJK 처리에 별도 코드 불필요
약점 — 기계의 시간을 쓴다
  • 실행 속도 — C/C++보다 느림 (수십 배까지)
  • 런타임 타입 오류 — 정적 검사 없음 (타입 힌트로 완화)
  • GIL — 순수 파이썬 멀티스레드 CPU 병렬 제한
  • 배포 무게 — 인터프리터+의존성 동봉

한 줄 요약 — Python은 "사람의 시간"을, C/C++는 "기계의 시간"을 아끼도록 최적화된 언어다.

02 · 의존성 & 버전

고정(pinning) vs 범위(하한·상한)

표기의미
==1.2.3정확히 그 버전 (고정/pinning)
>=1.2이상 (하한, lower bound)
<=1.2이하 (상한, upper bound)
>=1.2,<2.0범위
~=1.2.3호환 범위 (=1.2.*)
pyproject.toml vs requirements.txt
  • pyproject.toml — 프로젝트 정의서(레시피). 보통 느슨한 범위.
  • requirements.txt — 설치할 목록(장보기). 정확히 고정도 가능.
  • lock(uv.lock·poetry.lock) — 하위 의존성까지 정확 재현.

범위 = 자동 버그픽스 ↔ 깨질 위험 · 고정 = 재현성 ↔ 수동 갱신. requires-python=">=3.11"도 같은 개념(파이썬 버전 제약).

선언 없이, 값이 곧 타입

game.py
player = Hand.ROCK        # 상자에 넣으면 끝 — player는 Hand
computer = Hand(2)        # SCISSORS (숫자로 되찾기)
diff = (player - computer + 3) % 3   # diff는 int

# 타입 힌트 — 붙여도 되고 안 붙여도 되는 안전 표시

_ALIASES: dict[str, Hand] = { ... }
  • Java의 Hand player = ... 같은 타입 선언이 없다 — 대입이 곧 생성.

03 · 쏟아지는 질문들

변수? 상수? 일급?

변수(變數) ↔ 상수(常數)
  • 변수 = 變(변할 변)·數(셈 수) — 다시 대입해 바꿀 수 있는 이름.
  • 상수 = 常(항상 상) — C++ const·Java final이 재대입 금지를 컴파일러로 강제.
  • 파이썬에 const가 없는 이유: ①철학 — "우리는 모두 책임질 줄 아는 어른"(강제 대신 대문자 관례) ②변수가 상자가 아니라 이름표라 고정 장치가 없음(Final 힌트는 도구가 검사) ③대신 값을 불변으로 — tuple·frozen dataclass·Enum 멤버.
일급(first-class) = 시민권 등급
  • 일급 시민 = 차별 없이 모든 권리를 가진 시민. 값의 4대 권리 — ①변수에 담기 ②인자로 넘기기 ③반환하기 ④자료구조에 넣기.
  • 일급 함수 = 함수도 그 권리를 전부 누리는 언어에서의 함수 — 함수의 종류가 아니라 언어의 대우에 대한 말.
  • 이급(second-class)도 실존 용어 — C의 함수(포인터 우회만), 파이썬의 if 문(값이 아니라 못 담음).
03 · 클래스

__init__ — 멤버 변수는 어디서 "태어나나"

C++ — 멤버 선언 = 객체의 메모리 배치도
student.cpp
class Student {
    int    age;   // ① 선언 — 객체마다 이 "칸"이
    double gpa;   //    실제 메모리에 잡힌다
 public:
    Student(int a, double g)
        : age(a), gpa(g) {}   // ② 생성자 = 칸 채우기
};

Student s1(20, 3.8);  // s1: [age=20][gpa=3.8]
Student s2(24, 4.1);  // s2: 별도 메모리 — 독립
// sizeof(Student) = 칸들의 합(+정렬) — 배치가 눈에 보임
Python — 대입의 순간이 곧 탄생
student.py
class Student:
    # 선언부가 없다 — __init__의 대입이
    # ①선언 + ②초기화를 동시에 한다
    def __init__(self, age, gpa):
        self.age = age   # ← 이 순간 칸이 생김
        self.gpa = gpa

s1 = Student(20, 3.8)
s1.__dict__   # {'age': 20, 'gpa': 3.8}
# 인스턴스 = 이름→값 딕셔너리. s2는 각자의 것.
  • C++는 컴파일 때 고정 배치, Python은 실행 중 __dict__에 추가 — 선언 없이 대입만으로 되는 이유.
  • __init__ = 멤버 선언+생성자를 한 곳에. self = 그 인스턴스(명시된 this). 이 repo의 self._rng도 같은 원리.

던더(dunder) = Double UNDERscore의 준말("매직 메서드"). 밑줄 두 개 = "내가 아니라 파이썬 문법이 부르는 자리"라는 격리 표시다.

문법 → 실제로 불리는 던더

던더를 정의하면 내 타입을 언어 문법에 끼워 넣는다 — dataclass가 __eq__를 만들어 주면 ==가 그냥 동작하는 이유.

같은 문제, 세 언어의 해법
PythonC++Java
방식이름 규약 — 문법이 자동 호출operator 키워드오버로딩 없음 — 직접 호출

Java는 ==가 참조 비교로 고정, C++는 별도 선언 문법 필요 — Python은 이름만 맞추면 끝. "끼워 넣기" 비용이 가장 낮다.

던더언제 불리나 · 이 repo에서

03 · 예외 처리

우리도 "라벨을 붙이는 계층"이다

try/except를 쓰는 순간, 01장의 그 이야기가 내 코드의 일이 된다 — 내가 붙인(또는 삼킨) 라벨이 다음 디버거가 볼 전부다.

이 repo의 실제 코드
cli.py · web.py
# 예상한 상황만, 좁게 잡는다
try:
    result = game.play(user_input)
except ValueError as exc:      # 잘못된 손 입력
    print(f"오류: {exc}", file=sys.stderr)
    return 1

# 라벨을 바꿀 땐 인과 사슬을 보존
except ValueError as exc:
    raise HTTPException(400, str(exc)) from exc
    # from exc = "이 400의 원인은 저 ValueError"
명확한 자기 분석 4단계
  • ① 나는 지금 뭘 하고 있는가 — 이 블록이 시도하는 일.
  • ② 어떤 상황이 발생할 수 있는가 — 예상 가능한 실패 목록.
  • ③ 각각에 내가 뭘 할 수 있는가 — 복구·기본값·재시도. 못 하면 잡지 말고 전파.
  • ④ 조용히 삼키지 않는다 — 맨몸 except:는 다음 사람에게 'oom' 같은 오리무중 라벨을 남기는 것.

좋은 예외 처리 = 좋은 라벨 달기. 분석한 만큼만 잡고, 나머지는 정직하게 흘려보내라.

04 · 언어 비교

같은 게임, 다른 무게

판정 로직은 세 언어가 사실상 동일하다. 차이는 "그 로직을 감싸는 언어의 껍질"에서 난다.

Java · Spring

구조를 직접 세운다

계층·어노테이션·빌드 설정. 대규모 협업에 강하지만 무겁다.

C++20

기계에 가깝다

헤더/구현 분리, 소유권(unique_ptr) 관리. 빠르지만 손이 많이 간다.

Python

구조를 언어가 준다

IntEnum·dataclass·일급 함수가 스캐폴딩을 대신한다.

판정 공식은 셋 다 똑같다: diff = (player − computer + 3) % 3 → 0 무승부 · 1 승 · 2 패

04 · 비교의 원칙

한 줄만 보면 착시가 생긴다

판정식 한 줄 — 거의 같아 보인다
// Java
int diff = (player.code() - computer.code() + 3) % 3;

# Python
diff = (player - computer + 3) % 3
그 한 줄이 "되려면" 새로 생겨야 하는 것
Hand.java 에 추가될 코드
private final int code;              // ① 정수 필드
Hand(int code){ this.code = code; }  // ② 생성자
public int code(){ return code; }    // ③ 접근자
public static Hand fromCode(int c)   // ④ 역조회…

player.code() 한 조각 뒤에 필드·생성자·접근자·역조회가 딸려온다. Python은 IntEnum이라 이 네 가지가 전부 불필요 → 다음 장에서 34줄 전체를 본다.

Java — Hand.java 34줄 (전체)

        for (Hand h : values())      // 역조회 직접

Python — 전부 (5줄)
game.py
class Hand(IntEnum):

  • Hand.ROCK.name → "ROCK" — 공짜
  • Hand.ROCK - Hand.PAPER 산술 — 공짜
  • C++는 to_code/from_code/to_string 직접 구현 41줄

04 · 전체 구현 ③

다국어 파서 — 정규화 12줄 vs 한 줄

C++ — 공백 제거 + 소문자화를 손수
hand_parser.cpp (발췌)
std::string normalize(const std::string& in) {
  const char* ws = " \t\r\n";
  std::size_t b = in.find_first_not_of(ws);
  if (b == std::string::npos) return "";
  std::size_t e = in.find_last_not_of(ws);
  std::string s = in.substr(b, e - b + 1);
  std::transform(s.begin(), s.end(), s.begin(),
    [](unsigned char c){ return std::tolower(c); });
  return s;
}   // + 헤더 파일 별도 → 2파일 86줄
Python — 파서 전체 (~12줄)
game.py
def parse_hand(text: str) -> Hand:
    if text is None:

    hand = _ALIASES.get(text.strip().lower())
    if hand is None:
        raise ValueError(f"알 수 없는 손: {text!r}")

  • 헤더/구현 분리 없음 — 1파일

실습 01 · 바이너리 직접 실행

"설치"와 "직접 실행"은 다르다

python.org 에서 zip 하나를 받아 압축만 풀고, 그 안의 python.exe로 가위바위보를 돌린다 — 인터프리터도 결국 폴더 안의 실행 파일임을 확인하는 실습.

설치판 (installer)embeddable (zip) ← 이 실습
받는 것설치 프로그램(.exe)압축 파일(.zip)
하는 일PATH 등록 · py 런처 · 레지스트리압축 해제가 끝 — 아무것도 등록 안 함
실행어디서든 python, py폴더 안 python.exe를 경로로 직접
용도개발 환경앱에 파이썬 동봉 배포 · 격리 실행

이 PC에서 py --list가 되는 건 설치판이 등록해 둔 편의 — 이 실습은 그 도움 없이 바이너리를 직접 다룬다.

1.2.3정확히 고정 (==)

            
          

          
캐럿이 poetry add의 기본 —
            "메이저가 안 바뀌는 선에서 최신"이라는 범위 전략이다.

실습 05 · conda

      
만들고, 쓰고, 통째로 지우기

      

        

          
Miniconda 설치 후 — 명령은 두 OS 동일


# Windows: Anaconda Prompt / macOS: 터미널
conda create -n rps python=3.11  # 이름으로 환경 생성
conda activate rps               # (rps) 표시

pip install -e .                 # conda 안에서 pip도 OK
rps 바위
# 당신: ROCK / 컴퓨터: PAPER -> LOSE

conda env list                   # 내 환경 목록
conda deactivate
conda remove -n rps --all        # 통째로 정리

        

        

          
    
            
  • 설치는 Miniconda(최소판) 권장 — Windows 인스톨러 / macOS pkg 또는
              brew install --cask miniconda.
    • 
            
  • 원칙: conda에 있으면 conda로, 없는 것만 pip으로 — 네이티브 의존성 충돌을 줄이는 관례.
    • 
            
  • GPU 세팅도 같은 흐름: conda create -n torch python=3.11 →
              conda install pytorch pytorch-cuda …
    • 
            
  • 이름 기반이라 프로젝트 폴더 밖에서도 환경을 재사용할 수 있다.

실습 06 · 웹 서버 배포

      
개발: uvicorn으로 띄우고, 불러본다

      

        

          
실측 출력 — Windows·macOS 동일


uv sync --extra web
uv run uvicorn rps.web:app --reload

    • 
            
  • 브라우저로 /docs 열면 자동 문서(Swagger) — Pydantic 모델 덕.

  • 종료는 서버 터미널에서 Ctrl+C.

실습 06 · 웹 서버 배포

      
운영: gunicorn — 어디서 돌리나

      

        

          
macOS·리눅스는 그대로 / Windows는 WSL·Docker


# macOS·리눅스 — 워커 4개로 바로
gunicorn -k uvicorn.workers.UvicornWorker \
         rps.web:app --workers 4

# Windows에서 직접 실행하면? (실측)
# ModuleNotFoundError: No module named 'fcntl'
# → 유닉스 전용 모듈. WSL 또는 Docker로:
FROM python:3.13-slim
RUN pip install -e ".[web]"
CMD ["gunicorn","-k","uvicorn.workers.UvicornWorker",
     "rps.web:app","--bind","0.0.0.0:8000"]

        

        

          
    
            
  • 운영은 워커 여러 개 = 동시 처리↑ + 한 워커가 죽어도 서비스 유지(장애 격리).
    • 
            
  • gunicorn은 유닉스용 — 01장 "같은 코드도 OS에 따라 다르다"의 실전 사례.
    • 
            
  • 실제 배포는 대개 리눅스 컨테이너 — 그래서 개발도 WSL·Docker로 맞추는 흐름.
    • 
            
  • Windows 로컬 개발은 uvicorn(--workers 가능)으로 충분.
    •
