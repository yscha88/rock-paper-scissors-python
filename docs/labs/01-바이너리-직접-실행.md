# 랩 01 — 파이썬 바이너리를 직접 받아서 실행하기

**목표**: 설치 프로그램 없이, python.org 에서 **zip 하나를 받아 압축만 풀고** 그 안의
`python.exe` 로 가위바위보를 실행한다. "인터프리터도 결국 폴더 안의 실행 파일(바이너리)일
뿐"이라는 걸 몸으로 확인하는 랩이다.

> 소요 시간 10분 · 준비물: PowerShell, 인터넷 (zip 다운로드 약 12MB)

## 0. 개념 — "설치"와 "바이너리 직접 실행"은 다르다

파이썬을 쓰는 두 가지 방식:

| | 설치판 (installer) | **embeddable (zip)** ← 이 랩 |
|---|---|---|
| 받는 것 | 설치 프로그램(.exe) | **압축 파일(.zip)** |
| 하는 일 | PATH 등록, `py` 런처, 레지스트리 등록까지 | **압축 해제가 끝** — 아무것도 등록 안 함 |
| 실행 | 어디서든 `python`, `py` | **폴더 안의 `python.exe` 를 경로로 직접** |
| 용도 | 개발 환경 | 앱에 파이썬을 **동봉**해 배포, 격리 실행 |

이 PC 에서 `py --list` 가 되는 건 **설치판이 이미 깔려 PATH·레지스트리에 등록돼 있기
때문**입니다. 이 랩은 그 도움 없이, 바이너리를 직접 받아 실행합니다.

## 1. embeddable zip 다운로드 & 압축 해제

```powershell
cd $env:TEMP
curl.exe -L -o pyembed.zip https://www.python.org/ftp/python/3.13.0/python-3.13.0-embed-amd64.zip
Expand-Archive pyembed.zip -DestinationPath C:\py-embed
```

- `embed`(embeddable) = "다른 프로그램에 **끼워 넣을 수 있는**" 최소 배포판이라는 뜻.
- 설치 과정이 **없습니다**. 압축을 푼 순간 끝.

폴더 안을 보면:
```powershell
Get-ChildItem C:\py-embed | Select-Object Name
```
```
python.exe          ← 인터프리터 본체 (이게 "파이썬 바이너리")
python313.dll       ← 인터프리터의 실제 구현
python313.zip       ← 표준 라이브러리 (압축된 채로 씀)
python313._pth      ← 모듈 탐색 경로 설정 파일 (잠시 뒤 등장)
*.pyd               ← C 로 짠 표준 모듈들 (_socket, _sqlite3 ...)
```

## 2. 바로 실행해 보기

```powershell
C:\py-embed\python.exe --version
# Python 3.13.0
C:\py-embed\python.exe -c "print('hello from raw binary')"
```

PATH 등록도, `py` 런처도 없으니 **전체 경로로 직접** 부릅니다. 이게 "바이너리 직접 실행"입니다.

## 3. 이 바이너리로 가위바위보 돌리기 — 함정 포함

설치판에서 하던 대로 `PYTHONPATH` 를 줘 보면:

```powershell
cd C:\repo\yscha88\rock-paper-scissors-python
$env:PYTHONPATH = "src"
C:\py-embed\python.exe -m rps 바위
# python.exe: No module named rps        ← 실패!
```

**실패합니다.** embeddable 은 폴더 안의 **`python313._pth` 파일이 모듈 탐색 경로를 잠가**
두기 때문에, `PYTHONPATH` 환경변수를 **무시**합니다(격리 실행이 목적이라서). 내용을 보면:

```
python313.zip     ← 표준 라이브러리
.                 ← python.exe 가 있는 폴더
```

이 두 곳**만** 뒤집니다. 해법은 `._pth` 에 우리 코드 경로를 **직접 한 줄 추가**하는 것:

```powershell
Add-Content C:\py-embed\python313._pth "C:\repo\yscha88\rock-paper-scissors-python\src"
C:\py-embed\python.exe -m rps 바위
# 당신: ROCK / 컴퓨터: ROCK -> DRAW      ← 성공!
```

> 방금 한 것이 사실 이 랩의 핵심 수업입니다 — **인터프리터가 모듈을 찾는 경로(sys.path)는
> 설정 가능한 목록**이고, 설치판은 `PYTHONPATH`·site-packages 로, embeddable 은 `._pth` 로
> 그 목록을 관리합니다. "모듈을 못 찾는다(`No module named ...`)" 오류의 정체가 바로 이 목록 문제입니다.

## 4. 왜 이게 되는가

- 가위바위보 핵심 로직은 `secrets`·`enum`·`dataclasses` 등 **표준 라이브러리만** 씁니다.
  표준 라이브러리는 `python313.zip` 에 통째로 들어 있으니, **바이너리 폴더 하나로 완결**됩니다.
- 반대로 `fastapi` 같은 **외부 패키지**가 필요해지는 순간, "어디에 설치하고 어떻게 격리하나"
  문제가 생깁니다 → 그게 다음 랩(가상환경)의 주제입니다.
- 실무에서 embeddable 은 **앱에 파이썬을 동봉**할 때 쓰입니다(사용자 PC 에 파이썬이 없어도
  내 앱이 돌게). "Python 은 배포가 무겁다"(→ [01-cs-기초](../01-cs-기초.md))의 실전 대응이기도 합니다.

## 5. 참고 — 설치판은 뭐가 다른가

이 PC 에는 설치판도 여러 개 깔려 있습니다. 설치판은 `py` 런처가 버전을 관리해 줍니다:

```powershell
py --list          # 설치판 목록 (설치기가 등록해 둔 것)
py -3.11 --version # 버전 골라 실행
```

| | embeddable (이 랩) | 설치판 |
|---|---|---|
| 실행 | 전체 경로로 직접 | `python`, `py` (PATH 등록됨) |
| 모듈 경로 | `._pth` 로 잠금, PYTHONPATH 무시 | PYTHONPATH·site-packages 사용 |
| pip | 기본 없음 | 있음 |

## 6. macOS 에서는 — embeddable 대신 "단독 빌드"

embeddable zip 은 **Windows 전용** 배포판입니다. macOS 에서 같은 "압축 풀면 끝" 경험은
**python-build-standalone**(uv 가 내부에서 쓰는 단독 빌드)으로 합니다:

```bash
# Apple Silicon 기준 (Intel Mac 은 aarch64 → x86_64 로 바꾼 파일)
curl -LO https://github.com/astral-sh/python-build-standalone/releases/download/20250106/cpython-3.13.1+20250106-aarch64-apple-darwin-install_only.tar.gz
tar -xzf cpython-*.tar.gz            # python/ 폴더가 생김 — "설치"가 아님
./python/bin/python3 --version       # Python 3.13.1

cd ~/repo/rock-paper-scissors-python # 프로젝트 클론 위치로
PYTHONPATH=src <압축푼경로>/python/bin/python3 -m rps 바위
```

- 이 빌드에는 `._pth` 잠금이 없어 **`PYTHONPATH` 가 그대로 동작**합니다 — Windows embeddable
  과의 차이이자, "모듈 탐색 경로는 배포판마다 설정이 다르다"는 §3 교훈의 반복입니다.
- 정리도 같습니다 — 풀었던 `python/` 폴더만 지우면 끝.

## 7. 정리

```powershell
Remove-Item -Recurse -Force C:\py-embed
Remove-Item $env:TEMP\pyembed.zip
```

압축 풀었던 폴더만 지우면 흔적 없이 끝 — **레지스트리에 아무것도 안 남겼으니까요.**
이것도 embeddable 의 특징입니다.
