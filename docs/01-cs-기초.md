# 01. 코딩을 위한 CS 기초 — 소스코드가 실행되기까지

> **코딩이 처음이어도** 읽을 수 있게, 용어를 하나하나 풀어 씁니다. 가위바위보를 예로
> C/C++ · Java · Python 세 언어의 공통점과, 그중 Python 의 위치·강점·약점을 봅니다.

## 0. 아주 기본부터 — 이 문서에 계속 나올 단어들

| 단어 | 쉽게 말하면 |
|---|---|
| **프로그램(program)** | 컴퓨터에게 시키는 "할 일 목록". 우리가 만드는 가위바위보도 프로그램. |
| **소스코드(source code)** | 사람이 읽고 쓰는 그 할 일 목록의 **글**. `game.py` 같은 파일에 담긴다. |
| **CPU** | 컴퓨터의 **두뇌**. 실제 계산과 실행을 담당한다. |
| **메모리(memory)** | 실행 중인 값과 명령이 올라가는 **작업대**(RAM). |
| **실행(run)** | 소스코드에 적힌 대로 컴퓨터가 **실제로 일하게** 하는 것. |
| **이진/비트(binary/bit)** | 컴퓨터가 아는 유일한 글자, **0과 1**. 모든 것이 결국 0/1 로 표현된다. |
| **기계어(machine code)** | CPU 가 직접 알아듣는, 0/1 로 된 명령어. 사람은 거의 못 읽는다. |
| **어셈블리(assembly)** | 그 기계어를 **사람이 읽을 수 있게 적은 표기법**. 기계어와 거의 1:1 대응. |

> 비유: **소스코드는 "한국어로 쓴 요리 레시피"**, **CPU 는 한국어를 못 읽고 "기계어"만 아는 요리사**
> 입니다. 그래서 레시피(코드)를 요리사(CPU)가 아는 말로 **번역**해 줘야 요리(실행)가 됩니다.

## 1. 우리가 쓴 코드는 CPU 가 못 읽는다

우리가 쓴 `diff = (player - computer + 3) % 3` 같은 줄은 **사람을 위한 표현**입니다. CPU 는 이걸
그대로 못 읽고, **기계어(0/1)** 로 바뀌어야 실행됩니다. 이 "번역"을 **언제·어떻게** 하느냐가
프로그래밍 언어를 가르는 가장 큰 차이입니다.

```
소스코드(.py/.java/.cpp)  ──[ 번역 ]──▶  기계어(0/1)  ──▶  CPU 가 실행
   사람이 읽는 글                              요리사가 아는 말        요리 완성
```

### AI 시대에는 층이 하나 더 얹혔다

요즘은 사람이 소스코드를 직접 쓰는 대신 **프롬프트**를 쓰고, AI 에이전트가 코딩하는 경우가
많습니다. 그러면 그림이 이렇게 됩니다:

```
프롬프트(사람의 말)  ──[ AI 가 코딩 ]──▶  소스코드  ──[ 번역 ]──▶  기계어(0/1)  ──▶  CPU/GPU 실행
```

**번역 층이 하나 늘었을 뿐, 각 층의 번역은 여전히 틀릴 수 있습니다.** AI 가 만든 소스코드에
`age++` 같은 논리 버그(슬라이드 3)가 섞여도 기계는 군말 없이 그대로 실행합니다. 그래서
**중간층인 소스코드를 읽을 줄 아는 사람만이 그 번역이 맞았는지 검증**할 수 있습니다 — 프롬프트
시대에 코딩을 배우는 이유가 이 그림 안에 있습니다. (참고: AI 모델 자체의 학습·추론처럼 대규모
병렬 계산은 CPU 가 아니라 **GPU** — 수천 개의 단순 계산기를 묶은 칩 — 가 담당합니다.)

## 2. 번역을 "언제" 하느냐 — 3가지 방식

레시피를 요리사에게 전하는 방법이 세 가지라고 생각하면 쉽습니다.

| 방식 | 비유 | 대표 언어 | 실행 속도 |
|---|---|---|---|
| **컴파일 (네이티브)** | 요리 **전에** 레시피 전체를 기계어로 번역해 "완성본"을 만들어 둠 | C, C++, Go, Rust | 가장 빠름 |
| **바이트코드 + 가상머신** | 중간 언어로 미리 번역 → **가상의 요리사(VM)** 가 실행 | Java, C# | 중간 |
| **인터프리터** | 요리하면서 **한 줄씩 즉석 통역** | Python, Ruby, JS | 느림 |

몇 가지 용어를 더 풀면:

- **컴파일(compile)**: 소스코드를 미리 기계어로 바꾸는 것. 그 결과물이 **실행 파일**(`rps.exe`).
- **빌드(build)**: 소스코드를 실행 가능한 형태로 만드는 전체 과정(컴파일 + 묶기 등).
- **인터프리터(interpreter)**: 코드를 **실행하면서 그때그때 번역**해 주는 프로그램. Python 에서는
  `python` 이라는 프로그램이 그 역할을 한다.
- **가상머신(VM)**: 진짜 CPU 위에 얹힌 "가상의 실행기". Java 코드는 진짜 CPU 가 아니라
  **JVM**(자바 가상머신) 위에서 돈다.

> 이 셋은 칼같이 나뉘지 않습니다. 예를 들어 **Python** 도 사실 내부에서 코드를
> **바이트코드(`.pyc`)로 컴파일**한 뒤 실행합니다. Java 와의 진짜 차이는 "컴파일하느냐"가 아니라
> **언제, 얼마나 최적화해서** 번역하느냐입니다.

### 세 언어의 위치

- **C / C++** — 실행 **전에** 통째로 기계어로 번역(네이티브 컴파일). 실행 시 번역이 없어 **가장 빠르지만**,
  OS·CPU 가 바뀌면 다시 빌드해야 한다.
- **Java** — 미리 **바이트코드**로 번역해 두고, 실행할 때 **JVM** 이 돌린다. 자주 쓰는 부분은
  **JIT** 로 그 자리에서 기계어로 재번역해 빨라진다. "한 번 작성하면 어디서나 실행".
- **Python** — 빌드 단계 없이, `python` 이 코드를 **즉석에서 번역·실행**한다. 그래서
  **고치고 바로 돌려보는 속도가 가장 빠르지만**, 실행 자체는 셋 중 가장 느리다.

> **가위바위보로 보면**: C++ 는 `cmake --build` 로 `rps.exe` 를 **먼저 만든 뒤** 실행하고,
> Java 는 `gradlew build` 로 `.jar` 를 만들어 JVM 으로 돌립니다. Python 은 빌드 없이
> `python -m rps 바위` **한 줄이면 끝**입니다 — 이 "빌드 없음"이 Python 생산성의 큰 부분입니다.

## 3. 번역의 실제 단계 — "어셈블리"라는 중간역

"컴파일하면 기계어가 된다"는 사실 **여러 단계의 줄임말**입니다. C/C++ 기준으로 펼치면:

```
소스코드(.cpp)
  → [전처리기]   #include / #define 을 글자 그대로 펼침
  → [컴파일러]   어셈블리(.s) 생성        ← 사람이 읽을 수 있는 기계어 표기
  → [어셈블러]   목적 파일(.o / .obj)     ← 진짜 기계어(0/1) 조각
  → [링커]       목적 파일들 + 라이브러리를 이어붙여 실행 파일(rps.exe) 완성
```

### 어셈블리는 어떻게 생겼나

**어셈블리**는 기계어 명령 하나하나를 사람이 읽을 수 있는 이름(니모닉)으로 적은 것입니다.
판정식 `diff = (a - b + 3) % 3` 은 대략 이런 모습이 됩니다(x86-64, 개념 예시):

```asm
mov  eax, [a]      ; a 값을 레지스터(CPU 안의 임시 저장소)에 읽어 온다
sub  eax, [b]      ; b 를 뺀다
add  eax, 3        ; 3 을 더한다
mov  ecx, 3
cdq                ; 나눗셈 준비(부호 확장)
idiv ecx           ; 3 으로 나눈다 — 몫은 eax, "나머지는 edx" 에 남는다
mov  [diff], edx   ; 나머지를 diff 에 저장
```

한 줄짜리 파이썬 코드가 기계 입장에서는 이런 낱개 명령들의 나열입니다. **어셈블러**가 이
니모닉들을 0/1 opcode 로 바꾸면 그게 기계어입니다.

### CPU 는 기계어를 어떻게 "읽나" — 인출·해석·실행 사이클

CPU 는 마법이 아니라 단순한 반복 기계입니다. **인출(fetch) → 해석(decode) → 실행(execute)**:

1. **인출** — 메모리에서 다음 명령 하나를 가져온다 (어디까지 읽었는지는 **프로그램 카운터**가 기억).
2. **해석** — 그 0/1 이 무슨 명령인지(`sub`? `idiv`?) 해독한다.
3. **실행** — 계산하고, 결과를 레지스터나 메모리에 쓴다. 그리고 1로 돌아간다.

이 사이클이 **초당 수십억 번** 돕니다. Java 의 JVM, Python 의 인터프리터도 구조는 같습니다 —
바이트코드를 fetch-decode-execute 하는 **"소프트웨어로 만든 CPU"** 인 셈입니다.

### 링커는 왜 필요한가

우리 C++ 판도 파일이 여러 개입니다(`main.cpp`, `hand_parser.cpp`, `rock_paper_scissors.cpp`).
각 파일은 **따로** 목적 파일로 번역되는데, `main.cpp` 가 부르는 `parse_hand` 의 **실제 주소**는
그 시점엔 모릅니다. **링커**가 목적 파일들을 이어붙이면서 "이 호출은 저기 있는 함수"라고 주소를
연결해 줍니다. 연결할 상대를 못 찾으면 그 유명한 `undefined reference` 오류가 납니다.

## 4. 정적 타입 vs 동적 타입 — 값의 "종류"를 언제 확인하나

먼저 **타입(type)** 은 값의 **종류**입니다. `5` 는 정수(int), `"바위"` 는 문자열(str),
`Hand.ROCK` 은 손(Hand) — 이런 종류가 타입입니다. 이 타입을 **언제 확인**하느냐가 또 하나의 갈림길입니다.

> 비유: 택배 상자에 **"이건 컵"이라고 라벨을 미리 붙여두느냐(정적)**, 아니면 **열어봐야 뭔지
> 아느냐(동적)** 의 차이입니다.

- **정적 타입 (C/C++/Java)** — 변수의 타입을 **미리 선언**하고 **실행 전(컴파일 때) 검사**한다.
  "이 자리에는 손(Hand)만 들어간다"고 못 박아 두는 식. 실수를 실행 전에 잡아 **안전**하지만,
  타입을 일일이 써야 해서 **길어진다**.
- **동적 타입 (Python)** — 타입은 값에 붙어 있고 **실행 중에** 결정된다. 선언 없이 그냥 담는다.
  **짧고 유연**하지만, 타입 실수가 **실행 중에야** 드러날 수 있다.

```java
// Java: 앞에 타입(Hand, int)을 반드시 써야 한다
Hand player = Hand.ROCK;
int diff = (player.code() - computer.code() + 3) % 3;
```
```python
# Python: 타입을 안 써도 된다 — 값이 자기 타입을 안다
player = Hand.ROCK
diff = (player - computer + 3) % 3     # Hand 가 IntEnum 이라 바로 뺄셈
```

Python 도 원하면 **타입 힌트**(`player: Hand`)를 **덧붙일 수** 있습니다. 실행에는 영향을 주지 않고,
`mypy`·`pyright` 같은 **검사 도구**가 이 힌트를 읽어 실수를 미리 알려 줍니다. 즉 "동적이 기본,
정적은 선택"입니다(→ [03-python-문법](03-python-문법.md)).

한 걸음 더 나아가 **Pydantic** 같은 도구는 이 타입 힌트를 **실행 중에 검증**합니다. 밖에서 들어온
데이터(웹 요청·설정·JSON)가 선언한 타입과 맞는지 그때그때 확인해, 정적 타입이 약한 Python 에
**런타임 타입 안전**을 보태 줍니다(→ [03-python-문법](03-python-문법.md) 의 dataclass vs Pydantic).

## 5. 메모리 관리 — 다 쓴 그릇은 누가 치우나

**메모리(memory)** 는 프로그램이 값을 **잠깐 올려두는 작업대**입니다. 프로그램은 실행하면서 이
작업대를 계속 **빌리고(할당)**, 다 쓰면 **반납(해제)** 해야 합니다. 반납을 안 하면 작업대가 꽉 차
버립니다(이걸 **메모리 누수**라고 합니다). 이 "설거지"를 **누가** 하느냐가 언어마다 다릅니다.

| 언어 | 누가 치우나 | 개발자 부담 | 컴퓨터(런타임) 부담 |
|---|---|---|---|
| C | **직접** (`malloc` 으로 빌리고 `free` 로 반납) | **높음** — 깜빡하면 누수, 두 번 치우면 오류 | 낮음 — 정확히 관리하면 최소 |
| C++ | **범위를 벗어나면 자동** (RAII / `unique_ptr`) | 중간 | 낮음 |
| Java | **가비지 컬렉터(GC)** 가 알아서 회수 | 낮음 | **높음** — GC 실행 비용 + 여유 메모리 필요 |
| Python | **참조 카운팅 + GC** 가 알아서 회수 | 낮음 | 중간 — 참조 카운팅은 즉시, 순환 참조만 GC |

- **가비지 컬렉터(GC, garbage collector)**: "이제 아무도 안 쓰는 값"을 찾아 **자동으로 치워 주는
  청소부** 입니다. Java·Python 에 내장돼 있어 개발자가 반납을 신경 쓰지 않아도 됩니다.

> C++ 판 가위바위보가 난수 공급원을 `std::unique_ptr<RandomSource>` 로 감싸 소유하는 것은,
> 바로 이 "누가 언제 치우나"를 코드로 정해 주기 위해서입니다. Python·Java 는 그냥 값을 변수에
> 담아 두면 청소부(GC)가 알아서 치우므로, 같은 코드에서 **소유권 관리 코드가 통째로 사라집니다**.

부담이 기계로 넘어간 **대가**도 있습니다 — GC 언어는 같은 일에 **메모리를 더 쓰는 경향**이 있고,
GC 가 제때 안 비워 주면 낭비가 쌓이며, 회수가 도는 동안 **일시 멈춤(pause)** 이 생깁니다.
부담은 사라지는 것이 아니라 **개발자 ↔ 기계 사이를 이동**할 뿐입니다.

### GC 가 있어도 "관리"는 사라지지 않는다 — 규모가 커지면

작은 프로그램에서는 GC 를 잊고 살아도 됩니다. 그러나 Java 고급 과정으로 가면 **관리 대상이
바뀔 뿐, 관리 자체는 돌아옵니다**:

- **GC 알고리즘 선택** — JVM 의 GC 는 JDK 세대마다 다릅니다(JDK 8 의 Parallel → JDK 9+ 기본
  G1 → 저지연 ZGC·Shenandoah …). **처리량 우선이냐, 멈춤(지연) 최소화냐**의 트레이드오프라
  서비스 특성에 맞춰 골라야 합니다.
- **힙 튜닝** — `-Xmx`/`-Xms`, 세대(young/old) 크기, GC 로그 분석(`jstat`·GC 로그) — 대량
  트래픽 서비스에서는 일상 업무입니다.
- **JVM 에서도 누수는 납니다** — GC 는 "참조가 끊긴 것"만 회수하기 때문에:
  - **살아 있는 참조 누수**: static 컬렉션·무한 캐시·해제 안 한 리스너 — GC 가 원리상 못 치웁니다.
  - **ClassLoader/Metaspace 누수**: 재배포를 반복하면 클래스가 안 내려가 Metaspace OOM (레거시 WAS 단골).
  - **ThreadLocal 누수**: 스레드풀에서 `remove()` 를 안 하면 스레드가 사는 한 같이 삽니다.
  - **네이티브/DirectByteBuffer 누수**: **힙 밖**이라 GC 시야 밖 — `-Xmx` 를 아무리 늘려도 안 잡힙니다.
- 진단도 별도의 기술입니다 — 힙 덤프(`jmap`/`jcmd`) → MAT 분석 → "**무엇이 참조를 쥐고 있나**" 추적.

> 요약: GC 는 "해제 코드"를 없애 주는 것이지 "메모리 책임"을 없애 주는 게 아닙니다. C 의
> `free()` 관리가 → **참조 수명 관리 + GC 튜닝**으로 모양만 바뀝니다(→ [04장의 "규모가
> 커지면"](04-언어-비교.md)). 아래 OOM 이 바로 그 관리가 실패했을 때 만나는 화면입니다.

### OOM — 이름부터가 함정이다

그 낭비가 한계에 닿으면 만나는 것이 **OOM(Out Of Memory)** 입니다. 이름 그대로 "메모리가
모자랐구나(램을 늘리자)"라고 읽기 쉽지만 — **OOM 메시지만으로는 메모리가 정말 부족했는지 알 수
없습니다.** 특히 레거시 시스템에서는 "램은 넉넉한데 OOM"이 오히려 일상이었습니다.

1. **누수 — "못" 치우는 것**: 참조를 계속 쥐고 있으면 GC 는 "아직 쓰는 중"으로 판단해 치울 수
   없습니다(쌓기만 하는 전역 캐시가 전형). 부족해서가 아니라 **회수가 불가능**해서 죽습니다.
2. **한도(limit) 도달**: JVM `-Xmx`, 컨테이너(cgroup) memory limit — **서버 램이 남아돌아도**
   설정한 상한을 넘으면 OOM 입니다. "램은 널널한데 왜 죽지?"의 정체.
3. **OOM Killer — 죽임 "당하는" 것**: 리눅스 커널이 메모리가 몰리면 프로세스를 골라 강제
   종료합니다. 쿠버네티스의 `OOMKilled`(exit 137)가 이것 — 앱 내부 오류가 아니라 **바깥에서
   죽인 것**입니다.
4. **주소 공간 고갈·단편화 (레거시 단골)**: 32비트 프로세스는 주소 공간이 2~4GB 뿐이라 **물리
   램이 아무리 남아도** 주소 공간이 바닥나면 OOM 이었고, 큰 덩어리 할당은 **단편화** 때문에 빈
   총량이 충분해도 실패했습니다. 64비트 이행으로 드물어졌을 뿐, 레거시에서는 상식이었습니다.

그래서 레거시 세대는 OOM 을 보면 반사적으로 **확인 절차**를 밟았습니다 — ① 정말 물리 부족?
(`free`/`top`/모니터링 그래프) ② 프로세스 한도는? (`-Xmx`·`ulimit`·컨테이너 limit) ③ 무엇이
안 치워지나? (힙 덤프·객체 추이) ④ 누가 죽였나? (exit code·`dmesg` 의 oom-killer 로그).

### 에러명은 "사인(死因)"이지 "범인"이 아니다

실전 사례 — VS Code 창이 `reason: 'oom', code: 536870904` 로 죽는 유명한 크래시는 확장 누수,
대용량 파일, 심지어 **특정 코딩 폰트 + 합자(ligature)** 조합(CommitMono, PragmataPro Mono
Liga — 실제 확인된 사례)으로도 발생합니다. 왜 전부 "oom" 한 단어로 표시될까요?

- 수많은 원인(누수·폭주 루프·렌더링 버그)이 전부 **"메모리 고갈"이라는 같은 종착지**로
  흘러듭니다. 에러명은 그 **마지막 순간의 분류(사인)** 일 뿐, 범인(근본 원인)의 이름이 아닙니다.
- 부모 프로세스(VS Code 본체)는 죽은 자식(렌더러)의 **exit code 만 보고** 라벨을 붙입니다.
  'oom' 은 "이 프로세스가 메모리 계열 사유로 종료됨"이라는 **코드 분류**입니다.
- 그래서 같은 메시지 아래 **서로 다른 원인을 고친 해결책 수십 개**가 모입니다 — 정보가 넘치는
  시대에 근본 원인 찾기가 오히려 어려워지는 이유입니다.

실제로 이 크래시의 공식 이슈 중 하나(microsoft/vscode#226484)는 작성자가 확장 전체 제거·캐시
삭제·하드웨어 가속 비활성화·백신 예외·페이징 조정까지 **8가지를 시도하고도** 재현 조건을 좁히지
못해 **원인 미규명으로 닫혔습니다** — 증상 층에서 "해결책 룰렛"을 돌리는 것의 한계를 보여 주는
표본입니다.

반면 같은 계열의 **#239541** ("Render process crash due to OOM when using CommitMono font")은
역추적에 **성공**했습니다. 보고자의 결정적 단서는 "이전 버전(1.95.3)으로 돌아가면 안 죽는다"는
**버전 이등분**("무엇이 바뀌었나")이었고, 이슈에는 `terminal-ligatures`·`upstream` 라벨이 붙어
근본 원인이 VS Code 자신이 아니라 **업스트림(Chromium)의 합자 렌더링**으로 좁혀져 그쪽에서
수정됐습니다. #226484 의 한 코멘트(2025-05)도 같은 방식으로 **stickyScroll 설정 + PragmataPro
Mono Liga 폰트**의 조합을 찾아냈습니다 — 설정을 끄거나 **합자 없는 변형 폰트**로 바꾸면 사라진다
는 **A/B 격리**로요. 그리고 "터미널이 얼어요"(#241119) 같은 보고들은 `*duplicate` 로 묶였습니다
— **같은 범인이 oom / crashed / freeze 라는 서로 다른 사인**으로 보고되다 수렴한 것입니다.

**그럼 "폰트 때문"이라고 예상할 수 있었을까?** — 순방향 예측(라벨 → 범인)은 불가능합니다.
가능한 것은 **역추적**뿐입니다:

1. **재현 조건 격리** — 어느 파일/프로젝트에서만 죽는가? 그 파일에 뭐가 특별한가
   (CJK·합자가 많은 텍스트?).
2. **A/B 실험** — 폰트를 기본값으로 바꾸면? 합자(ligature) 설정을 끄면? 사라지면 범인 후보.
3. **크래시 덤프의 스택** — 죽는 순간 텍스트 셰이핑/렌더링 함수가 스택에 찍혀 있는가.
4. **메모리 프로파일** — 어떤 할당이 폭증하고 있는가.

이 역추적이 가능하려면 "렌더링 → 셰이핑 → 할당 → 라벨"이라는 **인과 사슬의 층들을 알고
있어야** 합니다 — 모르면 8가지 룰렛으로 끝납니다(#226484). 알면 버전 이등분과 A/B 격리로
업스트림까지 도달합니다(#239541). 그것이 이 문서가 층(§1~§6)을 하나씩 설명해 온 이유입니다.

마지막 반전도 기록해 둘 가치가 있습니다 — #239541 은 이후 봇이 "**최신 버전에서 수정됨**"을
선언했지만, 현장에서는 **최신 버전에서도 같은 'oom' 이 여전히 보고됩니다**. 모순이 아닙니다.
"fixed"는 **특정 인과 사슬 하나**(CommitMono 합자 → 렌더러 OOM)가 끊겼다는 뜻이지, 'oom' 이라는
**라벨이 다시 안 나온다는 보증이 아닙니다** — 종착지(메모리 고갈)로 흘러드는 다른 물길은
얼마든지 남아 있으니까요. 그래서 같은 라벨이 재발하면 "이미 고쳐졌다는데?"가 아니라 **새 사건으로
간주하고 역추적을 처음부터** 다시 해야 합니다. **라벨은 계속 살아 있고, 범인만 바뀝니다.**

이 이야기의 실천편이 [03장의 "예외 처리 — 우리도 라벨을 붙이는 계층이다"](03-python-문법.md)
입니다 — `except` 를 쓰는 순간 우리 자신이 라벨을 붙이는 계층이 되기 때문입니다.

### 예외 처리 계층이 라벨을 바꾼다 — "OOM ≠ nullptr" 의 단서 조항

"OOM=할당 실패 / nullptr=접근 위반"이라는 구분은 **기본 처리(핸들러가 없을 때)** 의 이야기입니다.
프로그램이 예외를 **어떻게 처리하느냐**에 따라 같은 원인이 다른 라벨을 답니다.

- **C++**: `new` 는 실패 시 `std::bad_alloc` **예외**를 던지고(안 잡으면 abort),
  `new(std::nothrow)` 는 **nullptr 를 반환**합니다 — 검사 없이 쓰면 segfault. C 의 `malloc` 도
  NULL 반환형. 즉 **같은 할당 실패가 abort 로도, nullptr 크래시로도** 나타납니다(OOM 의 위장).
- **Java**: `OutOfMemoryError` 는 던져지는 Error 라 `catch` 로 **삼킬 수도** 있습니다 — 그러면
  반쯤 초기화된 채 살아남았다가 **나중에 엉뚱한 NPE 로 재등장**합니다(라벨 변신).
- **Python**: `MemoryError` 역시 잡을 수 있는 예외입니다.
- 크래시 리포터(브라우저의 Crashpad 등)는 시그널/예외를 **가로채 분류**해 보고합니다 — 우리가
  보는 라벨은 이 계층의 산출물입니다.

**감별의 진짜 질문은 "이 라벨, 어느 계층이 붙였나?"** 입니다. exit **137**(밖에서 죽임 — OOM
계열) vs **139**(안에서 잘못된 주소 — SIGSEGV)는 그 첫 단서일 뿐입니다.

### 공유 런타임 시나리오 — 층이 얽힌 실전 원인분석

이런 상황을 가정해 봅시다: Windows 보안 패치와 함께 **WebView2**(Edge 와 같이 자동 업데이트되는
공유 웹 엔진)가 갱신됐고, 그 하위호환성 문제로 **노트북 제조사 번들 앱들**이 먼저 터지기
시작했으며, 비슷한 시기에 VS Code(원격 접속 창)도 'oom' 으로 죽는다면 — 원인분석은?

1. **"무엇이 바뀌었나?"부터.** 내 코드는 안 바뀌었는데 갑자기 터지면, 바뀐 것은 아래층입니다 —
   OS 패치, 공유 런타임의 **자동 업데이트(에버그린)**, 드라이버. 크래시 시작 시점과 업데이트
   타임라인을 대조합니다.
2. **상관관계 신호.** 서로 무관한 앱들이 **같은 시기에 같은 계열**로 터지면 공통
   의존성(공유 엔진·패치)을 의심합니다. 한 앱 안에서만 원인을 찾는 것(확장 끄기·캐시 삭제의
   반복)은 층을 잘못 짚은 것일 수 있습니다 — 번들 앱들의 동반 크래시가 오히려 힌트였습니다.
3. **버전 격리 실험.** 공유 런타임을 **고정(pinning)** 하거나 롤백해 재현 여부를 봅니다 —
   WebView2 는 Evergreen(자동) 대신 **Fixed Version** 배포가 가능합니다. 패치 제거, 다른 기기
   재현도 같은 원리입니다. (패키지 버전 고정/범위 개념이 **런타임 버전**에도 그대로 적용됩니다
   — [02장](02-python-환경.md).)
4. **라벨 해석.** 이 시나리오의 'oom' 은 (a) 호환성 어긋남 → 재시도·누수 폭주 → 진짜 고갈,
   (b) **힙 오염 감지 → 보안상 즉시 종료** — Chromium 계열 할당기는 오염(이중 해제·use-after-free
   흔적)을 발견하면 일부러 프로세스를 죽이며, 그 일부가 OOM 으로 분류됩니다. "부족해서"가 아니라
   "**믿을 수 없어서**" 죽였는데 라벨은 oom — (c) 공유 메모리 stale 실패. 어느 쪽인지는 크래시
   덤프·이벤트 뷰어·엔진 로그의 스택으로 좁힙니다.

**실측 사례** — 실제로 이런 시나리오를 조사해 보면, 앱이 보여 준 라벨('oom')과 달리 **윈도우
이벤트 뷰어**에는 다른 이야기가 적혀 있곤 합니다: WebView2 가 **보안 정책 위반(하위 호환성
미지원)** 으로 강제 종료되고, 감시자가 되살리고, 다시 죽는 **크래시 루프**가 제조사 레거시
번들 앱에서 주기적으로 반복된 기록. 여기서 교훈 두 가지 —

- **진짜 사인은 라벨을 붙인 계층보다 아래, "OS 의 장부"에 있다.** 앱의 에러 대화상자가 아니라
  이벤트 뷰어(`eventvwr`)·`dmesg`·저널 로그가 원본 기록입니다.
- **주기적인 죽음·부활은 "자동 재시작 루프"의 지문**입니다 — 서비스 매니저/감시 프로세스가
  계속 되살리고 있다는 뜻이고, 근본 원인을 못 고치면 로그에 주기성이 찍힙니다.

일반화하면: **실제 OOM 이 아닌데 'oom' 으로 분류되는 일은 비일비재합니다.** 크래시 리포터가
분류하지 못하는 급사를 메모리 계열로 뭉뚱그리거나, 보안 정책·외부 킬이 앱에게는 "렌더러가
갑자기 죽음"으로만 보이거나, 종료 코드 매핑이 부정확한 경우 — 라벨은 언제나 **"마지막에 남은
증거"이지 "판결"이 아닙니다.**

**"반납한 주소 자리에 다른 프로세스가 들어와 있다면?"** — 좋은 질문이지만, **프로세스끼리는
서로의 주소 공간에 들어올 수 없습니다.** 가상 메모리가 프로세스마다 독립된 주소 공간을 주고,
남의 영역 접근은 접근 위반으로 차단합니다(그게 segfault 의 존재 이유). 실제로 일어나는 것은 두
가지입니다 — **같은 프로세스 안**에서 해제된 자리가 다른 할당에 **재사용**되는 경우
(use-after-free: 즉시 터지지 않고 데이터가 미묘하게 오염된 뒤, 한참 뒤 엉뚱한 라벨로 터지는
대표 원인), 그리고 **공유 메모리 구간**(WebView2 같은 멀티프로세스 엔진이 host↔renderer 간에
실제로 공유)의 핸들이 엔진 업데이트로 **어긋나는(stale)** 경우. 어느 쪽이든 할당기 메타데이터가
오염되면 다음 할당이 실패하면서 — 라벨은 또 'oom' 이 됩니다.

## 6. 순서가 어긋나면 생기는 버그 — nullptr · undefined · NameError

번역·실행 모델(§2~§3)을 알아야 정체가 보이는 대표 버그가 있습니다 —
**"정의(초기화)보다 사용이 먼저 오는"** 문제입니다. 언어마다 증상 이름만 다릅니다.

- **C++ (컴파일·링크 시점)** — 서로 다른 파일(번역 단위)의 **전역 객체 초기화 순서는 보장되지
  않습니다**(악명 높은 *static initialization order fiasco*). A.cpp 의 전역 객체가 B.cpp 의 전역
  객체를 참조하는데 B 가 **아직 초기화되기 전**이면, **널 포인터(`nullptr`)나 쓰레기 값**을 만나
  크래시합니다. 링커에 라이브러리 **순서**를 잘못 주면 `undefined reference` 링크 오류도 납니다 —
  §3 의 "링커가 주소를 연결한다"가 실패한 것입니다.
- **Python (인터프리터 실행 시점)** — 인터프리터는 파일을 **위에서 아래로 실행하며 그때그때
  이름을 만듭니다**. 정의보다 호출이 먼저 오면:
  ```python
  play_game()                 # NameError: name 'play_game' is not defined
  def play_game(): ...        # 정의가 아래에 있으면 늦다 (모듈 최상위 기준)
  ```
  **순환 import**(A가 B를, B가 다시 A를 import)를 만들면 "반쯤 초기화된 모듈"을 보게 되어
  `AttributeError`·`ImportError` 가 납니다.
- **JavaScript** — `var x` 선언은 위로 끌어올려지지만(**호이스팅**) 값 대입은 제자리라서, 그 사이에
  읽으면 **`undefined`** 가 잡힙니다.

세 증상 — **널 포인터 / NameError / undefined** — 은 달라 보여도 뿌리가 같습니다:
**"이 이름(주소)이 쓰이는 시점에 준비되어 있는가?"** 언어의 실행 모델을 알면 이 부류의 버그가
한 가족으로 보이고, 어느 단계(컴파일? 링크? 실행?)에서 터졌는지로 원인을 좁힐 수 있습니다.
(3번 슬라이드의 `age++` 처럼 — **실행 모델을 모르면 못 잡는 버그**의 또 다른 예입니다.)

## 7. CS 이해가 부족하면 생길 수밖에 없는 디버깅들

레거시 세대가 "당연히 확인하던 것들"이 요즘은 두꺼운 추상화(프레임워크·클라우드·AI)에 가려
보이지 않게 되면서, 세대 간 격차가 점점 벌어지고 있습니다. 공통 구조는 하나입니다 —
**증상은 위층에서 보이지만, 원인은 아래층에 있다.**

| 증상 | 순진한 해석 | 진짜 원인 후보 | 필요한 아래층 지식 |
|---|---|---|---|
| `OOM` | "램 부족 → 늘리자" | 누수 · 한도(`-Xmx`/limit) · 주소 공간 · 단편화 · 힙 오염 | 메모리·GC (§5) |
| `nullptr`·`NameError`·`undefined` | "코드가 이상해요" | 초기화·링크·실행 **순서** | 번역·실행 모델 (§3·§6) |
| 한글이 `???` 로 깨짐 | "폰트 문제?" | **인코딩**(코드페이지 vs UTF-8) — C++판 `wmain` 사례 | 문자 표현 |
| `0.1 + 0.2 != 0.3` | "언어 버그?" | 부동소수점 **이진 표현** | 수 표현 |
| "어제는 됐는데 오늘 안 됨" | "컴퓨터 탓" | 의존성 **버전 범위**가 자동으로 새 버전을 당김 · 공유 런타임 자동 업데이트 | 고정 vs 범위 (02장) |
| "느려요" | "서버 늘리자" | GIL · GC pause · O(n²) 알고리즘 | 실행 모델·자료구조 |

도구가 좋아질수록 이 격차는 **더 벌어집니다** — AI 가 코드를 쓰는 시대(§1)일수록, **아래층을
읽는 능력이 곧 디버깅 능력**입니다. 이 저장소의 목표가 바로 그 "아래층 읽기"의 첫걸음입니다.

## 8. 그래도 세 언어는 형제다 — 공통 특성

번역 방식·타입·메모리에서 갈릴 뿐, 셋 다 뼈대는 같습니다. 그래서 한 언어를 배우면 나머지가 쉬워집니다.

- **변수 · 함수 · 제어 흐름**(조건 `if`, 반복 `for`)이라는 **기본 재료가 동일**하다.
- **타입/클래스로 데이터를 모델링**한다 — 이 repo 의 `Hand`(손), `Outcome`(승패), `GameResult`(결과)는
  세 언어 모두에 대응물이 있다(enum / enum / record·struct·dataclass).
- **같은 알고리즘이 그대로 옮겨진다** — 판정 공식 `(player - computer + 3) % 3` 은 세 파일에서
  글자까지 거의 같다. *언어가 달라도 "생각의 단위"는 공유된다*는 것이 핵심입니다.

그래서 반복문·함수 분리·예외 처리·테스트 같은 개념은 언어를 넘나들며 재사용됩니다. **Python 으로
개념을 익히면, Java/C++ 로 갈 때 "문법만" 다시 배우면** 됩니다.

## 9. Python 의 강점과 약점 (managed/컴파일 언어 대비)

### 강점 — "사람의 시간"을 아낀다
- **빌드가 없어 빠른 반복**: 고치고 바로 실행. `python -m rps 바위` 한 줄.
- **적은 코드량**: 동적 타입 + 풍부한 표준 라이브러리로 같은 일을 더 짧게. (이 repo 도메인:
  Java 5개·C++ 8개 파일 → **Python 1개 파일**)
- **배터리 포함(batteries included)**: `secrets`(보안 난수)·`enum`·`dataclasses` 가 **기본 내장** —
  아무것도 안 깔아도 핵심 로직이 완성된다.
- **어디에나 붙는 생태계**: 데이터·AI·웹·자동화 등 방대한 패키지.
- **유니코드 기본**: `str` 자체가 유니코드라 한글·일본어 처리에 추가 코드가 필요 없다
  (C++ 판이 Windows 에서 하던 인코딩 변환이 통째로 사라진다).

### 약점 — "기계의 시간"을 쓴다
- **실행 속도**: 한 줄씩 통역하듯 실행해 C/C++ 보다 느리다(수십 배까지). 무거운 계산엔 불리.
- **런타임 타입 오류**: 실행 전 타입 검사가 없어, 타입 실수가 **실행 중에** 터질 수 있다(타입 힌트로 완화).
- **GIL**: CPython 에는 **전역 인터프리터 락(GIL)** 이라는 잠금이 있어, 순수 파이썬 코드로
  여러 CPU 코어를 동시에 굴리는 **병렬 처리가 제한**된다.
- **배포가 무겁다**: 인터프리터 + 필요한 패키지를 함께 배포해야 한다(하나의 실행 파일이 기본은 아님).
- **정적 안전망이 약함**: 컴파일러가 잡아 주던 실수를, 사람이 **테스트·리뷰**로 더 메워야 한다.

> **한 줄 요약**: Python 은 **"사람의 시간"** 을, C/C++ 는 **"기계의 시간"** 을 아끼도록 최적화된
> 언어입니다. 가위바위보처럼 로직이 가볍고 빠른 수정이 중요한 문제에서는 Python 의 강점이 그대로
> 드러나고, 약점(속도)은 거의 문제가 되지 않습니다.
