# 05. 디버깅 — 라벨 너머의 범인 찾기

> 이론(01~04)을 실전 진단으로 잇는 장입니다. [01장 §5](01-cs-기초.md)에서 본 메모리·GC 의
> "낭비"가 한계에 닿으면 무엇을 만나는지, 에러 메시지(라벨)를 어떻게 읽어야 하는지,
> 그리고 CS 이해 없이는 잡을 수 없는 버그들의 카탈로그까지.

## OOM — 이름부터가 함정이다

그 낭비가 한계에 닿으면 만나는 것이 **OOM(Out Of Memory)** 입니다. 이름 그대로 "메모리가
모자랐구나(램을 늘리자)"라고 읽기 쉽지만 — **OOM 메시지만으로는 메모리가 정말 부족했는지 알 수
없습니다.** 특히 레거시 시스템에서는 "램은 넉넉한데 OOM"이 오히려 일상이었습니다.

1. **누수 — "못" 치우는 것**: 참조를 계속 쥐고 있으면 GC 는 "아직 쓰는 중"으로 판단해 치울 수
   없습니다(쌓기만 하는 전역 캐시가 전형). 부족해서가 아니라 **회수가 불가능**해서 죽습니다.
2. **한도(limit) 도달**: JVM `-Xmx`, 컨테이너(cgroup) memory limit — **서버 램이 남아돌아도**
   설정한 상한을 넘으면 OOM 입니다. "램은 널널한데 왜 죽지?"의 정체.
3. **OOM Killer — 죽임 "당하는" 것**: 리눅스 커널이 메모리가 몰리면 프로세스를 골라 강제
   종료합니다. 쿠버네티스의 `OOMKilled`(exit 137)가 이것 — 앱 내부 오류가 아니라 **바깥에서
   죽인 것**입니다.
4. **주소 공간 고갈·단편화 (레거시 단골)**: 32비트 프로세스는 주소 공간이 2~4GB 뿐이라 **물리
   램이 아무리 남아도** 주소 공간이 바닥나면 OOM 이었고, 큰 덩어리 할당은 **단편화** 때문에 빈
   총량이 충분해도 실패했습니다. 64비트 이행으로 드물어졌을 뿐, 레거시에서는 상식이었습니다.

그래서 레거시 세대는 OOM 을 보면 반사적으로 **확인 절차**를 밟았습니다 — ① 정말 물리 부족?
(`free`/`top`/모니터링 그래프) ② 프로세스 한도는? (`-Xmx`·`ulimit`·컨테이너 limit) ③ 무엇이
안 치워지나? (힙 덤프·객체 추이) ④ 누가 죽였나? (exit code·`dmesg` 의 oom-killer 로그).

## 에러명은 "사인(死因)"이지 "범인"이 아니다

실전 사례 — VS Code 창이 `reason: 'oom', code: 536870904` 로 죽는 유명한 크래시는 확장 누수,
대용량 파일, 심지어 **특정 코딩 폰트 + 합자(ligature)** 조합(CommitMono, PragmataPro Mono
Liga — 실제 확인된 사례)으로도 발생합니다. 왜 전부 "oom" 한 단어로 표시될까요?

- 수많은 원인(누수·폭주 루프·렌더링 버그)이 전부 **"메모리 고갈"이라는 같은 종착지**로
  흘러듭니다. 에러명은 그 **마지막 순간의 분류(사인)** 일 뿐, 범인(근본 원인)의 이름이 아닙니다.
- 부모 프로세스(VS Code 본체)는 죽은 자식(렌더러)의 **exit code 만 보고** 라벨을 붙입니다.
  'oom' 은 "이 프로세스가 메모리 계열 사유로 종료됨"이라는 **코드 분류**입니다.
- 그래서 같은 메시지 아래 **서로 다른 원인을 고친 해결책 수십 개**가 모입니다 — 정보가 넘치는
  시대에 근본 원인 찾기가 오히려 어려워지는 이유입니다.

실제로 이 크래시의 공식 이슈 중 하나(microsoft/vscode#226484)는 작성자가 확장 전체 제거·캐시
삭제·하드웨어 가속 비활성화·백신 예외·페이징 조정까지 **8가지를 시도하고도** 재현 조건을 좁히지
못해 **원인 미규명으로 닫혔습니다** — 증상 층에서 "해결책 룰렛"을 돌리는 것의 한계를 보여 주는
표본입니다.

반면 같은 계열의 **#239541** ("Render process crash due to OOM when using CommitMono font")은
역추적에 **성공**했습니다. 보고자의 결정적 단서는 "이전 버전(1.95.3)으로 돌아가면 안 죽는다"는
**버전 이등분**("무엇이 바뀌었나")이었고, 이슈에는 `terminal-ligatures`·`upstream` 라벨이 붙어
근본 원인이 VS Code 자신이 아니라 **업스트림(Chromium)의 합자 렌더링**으로 좁혀져 그쪽에서
수정됐습니다. #226484 의 한 코멘트(2025-05)도 같은 방식으로 **stickyScroll 설정 + PragmataPro
Mono Liga 폰트**의 조합을 찾아냈습니다 — 설정을 끄거나 **합자 없는 변형 폰트**로 바꾸면 사라진다
는 **A/B 격리**로요. 그리고 "터미널이 얼어요"(#241119) 같은 보고들은 `*duplicate` 로 묶였습니다
— **같은 범인이 oom / crashed / freeze 라는 서로 다른 사인**으로 보고되다 수렴한 것입니다.

**그럼 "폰트 때문"이라고 예상할 수 있었을까?** — 순방향 예측(라벨 → 범인)은 불가능합니다.
가능한 것은 **역추적**뿐입니다:

1. **재현 조건 격리** — 어느 파일/프로젝트에서만 죽는가? 그 파일에 뭐가 특별한가
   (CJK·합자가 많은 텍스트?).
2. **A/B 실험** — 폰트를 기본값으로 바꾸면? 합자(ligature) 설정을 끄면? 사라지면 범인 후보.
3. **크래시 덤프의 스택** — 죽는 순간 텍스트 셰이핑/렌더링 함수가 스택에 찍혀 있는가.
4. **메모리 프로파일** — 어떤 할당이 폭증하고 있는가.

이 역추적이 가능하려면 "렌더링 → 셰이핑 → 할당 → 라벨"이라는 **인과 사슬의 층들을 알고
있어야** 합니다 — 모르면 8가지 룰렛으로 끝납니다(#226484). 알면 버전 이등분과 A/B 격리로
업스트림까지 도달합니다(#239541). 그것이 이 문서가 층(§1~§6)을 하나씩 설명해 온 이유입니다.

마지막 반전도 기록해 둘 가치가 있습니다 — #239541 은 이후 봇이 "**최신 버전에서 수정됨**"을
선언했지만, 현장에서는 **최신 버전에서도 같은 'oom' 이 여전히 보고됩니다**. 모순이 아닙니다.
"fixed"는 **특정 인과 사슬 하나**(CommitMono 합자 → 렌더러 OOM)가 끊겼다는 뜻이지, 'oom' 이라는
**라벨이 다시 안 나온다는 보증이 아닙니다** — 종착지(메모리 고갈)로 흘러드는 다른 물길은
얼마든지 남아 있으니까요. 그래서 같은 라벨이 재발하면 "이미 고쳐졌다는데?"가 아니라 **새 사건으로
간주하고 역추적을 처음부터** 다시 해야 합니다. **라벨은 계속 살아 있고, 범인만 바뀝니다.**

이 이야기의 실천편이 [03장의 "예외 처리 — 우리도 라벨을 붙이는 계층이다"](03-python-문법.md)
입니다 — `except` 를 쓰는 순간 우리 자신이 라벨을 붙이는 계층이 되기 때문입니다.

## 예외 처리 계층이 라벨을 바꾼다 — "OOM ≠ nullptr" 의 단서 조항

"OOM=할당 실패 / nullptr=접근 위반"이라는 구분은 **기본 처리(핸들러가 없을 때)** 의 이야기입니다.
프로그램이 예외를 **어떻게 처리하느냐**에 따라 같은 원인이 다른 라벨을 답니다.

- **C++**: `new` 는 실패 시 `std::bad_alloc` **예외**를 던지고(안 잡으면 abort),
  `new(std::nothrow)` 는 **nullptr 를 반환**합니다 — 검사 없이 쓰면 segfault. C 의 `malloc` 도
  NULL 반환형. 즉 **같은 할당 실패가 abort 로도, nullptr 크래시로도** 나타납니다(OOM 의 위장).
- **Java**: `OutOfMemoryError` 는 던져지는 Error 라 `catch` 로 **삼킬 수도** 있습니다 — 그러면
  반쯤 초기화된 채 살아남았다가 **나중에 엉뚱한 NPE 로 재등장**합니다(라벨 변신).
- **Python**: `MemoryError` 역시 잡을 수 있는 예외입니다.
- 크래시 리포터(브라우저의 Crashpad 등)는 시그널/예외를 **가로채 분류**해 보고합니다 — 우리가
  보는 라벨은 이 계층의 산출물입니다.

**감별의 진짜 질문은 "이 라벨, 어느 계층이 붙였나?"** 입니다. exit **137**(밖에서 죽임 — OOM
계열) vs **139**(안에서 잘못된 주소 — SIGSEGV)는 그 첫 단서일 뿐입니다.

## 공유 런타임 시나리오 — 층이 얽힌 실전 원인분석

이런 상황을 가정해 봅시다: Windows 보안 패치와 함께 **WebView2**(Edge 와 같이 자동 업데이트되는
공유 웹 엔진)가 갱신됐고, 그 하위호환성 문제로 **노트북 제조사 번들 앱들**이 먼저 터지기
시작했으며, 비슷한 시기에 VS Code(원격 접속 창)도 'oom' 으로 죽는다면 — 원인분석은?

1. **"무엇이 바뀌었나?"부터.** 내 코드는 안 바뀌었는데 갑자기 터지면, 바뀐 것은 아래층입니다 —
   OS 패치, 공유 런타임의 **자동 업데이트(에버그린)**, 드라이버. 크래시 시작 시점과 업데이트
   타임라인을 대조합니다.
2. **상관관계 신호.** 서로 무관한 앱들이 **같은 시기에 같은 계열**로 터지면 공통
   의존성(공유 엔진·패치)을 의심합니다. 한 앱 안에서만 원인을 찾는 것(확장 끄기·캐시 삭제의
   반복)은 층을 잘못 짚은 것일 수 있습니다 — 번들 앱들의 동반 크래시가 오히려 힌트였습니다.
3. **버전 격리 실험.** 공유 런타임을 **고정(pinning)** 하거나 롤백해 재현 여부를 봅니다 —
   WebView2 는 Evergreen(자동) 대신 **Fixed Version** 배포가 가능합니다. 패치 제거, 다른 기기
   재현도 같은 원리입니다. (패키지 버전 고정/범위 개념이 **런타임 버전**에도 그대로 적용됩니다
   — [02장](02-python-환경.md).)
4. **라벨 해석.** 이 시나리오의 'oom' 은 (a) 호환성 어긋남 → 재시도·누수 폭주 → 진짜 고갈,
   (b) **힙 오염 감지 → 보안상 즉시 종료** — Chromium 계열 할당기는 오염(이중 해제·use-after-free
   흔적)을 발견하면 일부러 프로세스를 죽이며, 그 일부가 OOM 으로 분류됩니다. "부족해서"가 아니라
   "**믿을 수 없어서**" 죽였는데 라벨은 oom — (c) 공유 메모리 stale 실패. 어느 쪽인지는 크래시
   덤프·이벤트 뷰어·엔진 로그의 스택으로 좁힙니다.

**실측 사례** — 실제로 이런 시나리오를 조사해 보면, 앱이 보여 준 라벨('oom')과 달리 **윈도우
이벤트 뷰어**에는 다른 이야기가 적혀 있곤 합니다: WebView2 가 **보안 정책 위반(하위 호환성
미지원)** 으로 강제 종료되고, 감시자가 되살리고, 다시 죽는 **크래시 루프**가 제조사 레거시
번들 앱에서 주기적으로 반복된 기록. 여기서 교훈 두 가지 —

- **진짜 사인은 라벨을 붙인 계층보다 아래, "OS 의 장부"에 있다.** 앱의 에러 대화상자가 아니라
  이벤트 뷰어(`eventvwr`)·`dmesg`·저널 로그가 원본 기록입니다.
- **주기적인 죽음·부활은 "자동 재시작 루프"의 지문**입니다 — 서비스 매니저/감시 프로세스가
  계속 되살리고 있다는 뜻이고, 근본 원인을 못 고치면 로그에 주기성이 찍힙니다.

일반화하면: **실제 OOM 이 아닌데 'oom' 으로 분류되는 일은 비일비재합니다.** 크래시 리포터가
분류하지 못하는 급사를 메모리 계열로 뭉뚱그리거나, 보안 정책·외부 킬이 앱에게는 "렌더러가
갑자기 죽음"으로만 보이거나, 종료 코드 매핑이 부정확한 경우 — 라벨은 언제나 **"마지막에 남은
증거"이지 "판결"이 아닙니다.**

**"반납한 주소 자리에 다른 프로세스가 들어와 있다면?"** — 좋은 질문이지만, **프로세스끼리는
서로의 주소 공간에 들어올 수 없습니다.** 가상 메모리가 프로세스마다 독립된 주소 공간을 주고,
남의 영역 접근은 접근 위반으로 차단합니다(그게 segfault 의 존재 이유). 실제로 일어나는 것은 두
가지입니다 — **같은 프로세스 안**에서 해제된 자리가 다른 할당에 **재사용**되는 경우
(use-after-free: 즉시 터지지 않고 데이터가 미묘하게 오염된 뒤, 한참 뒤 엉뚱한 라벨로 터지는
대표 원인), 그리고 **공유 메모리 구간**(WebView2 같은 멀티프로세스 엔진이 host↔renderer 간에
실제로 공유)의 핸들이 엔진 업데이트로 **어긋나는(stale)** 경우. 어느 쪽이든 할당기 메타데이터가
오염되면 다음 할당이 실패하면서 — 라벨은 또 'oom' 이 됩니다.

## 순서가 어긋나면 생기는 버그 — nullptr · undefined · NameError

[01장](01-cs-기초.md)의 번역·실행 모델(§2~§3)을 알아야 정체가 보이는 대표 버그가 있습니다 —
**"정의(초기화)보다 사용이 먼저 오는"** 문제입니다. 언어마다 증상 이름만 다릅니다.

- **C++ (컴파일·링크 시점)** — 서로 다른 파일(번역 단위)의 **전역 객체 초기화 순서는 보장되지
  않습니다**(악명 높은 *static initialization order fiasco*). A.cpp 의 전역 객체가 B.cpp 의 전역
  객체를 참조하는데 B 가 **아직 초기화되기 전**이면, **널 포인터(`nullptr`)나 쓰레기 값**을 만나
  크래시합니다. 링커에 라이브러리 **순서**를 잘못 주면 `undefined reference` 링크 오류도 납니다 —
  01장 §3 의 "링커가 주소를 연결한다"가 실패한 것입니다.
- **Python (인터프리터 실행 시점)** — 인터프리터는 파일을 **위에서 아래로 실행하며 그때그때
  이름을 만듭니다**. 정의보다 호출이 먼저 오면:
  ```python
  play_game()                 # NameError: name 'play_game' is not defined
  def play_game(): ...        # 정의가 아래에 있으면 늦다 (모듈 최상위 기준)
  ```
  **순환 import**(A가 B를, B가 다시 A를 import)를 만들면 "반쯤 초기화된 모듈"을 보게 되어
  `AttributeError`·`ImportError` 가 납니다.
- **JavaScript** — `var x` 선언은 위로 끌어올려지지만(**호이스팅**) 값 대입은 제자리라서, 그 사이에
  읽으면 **`undefined`** 가 잡힙니다.

세 증상 — **널 포인터 / NameError / undefined** — 은 달라 보여도 뿌리가 같습니다:
**"이 이름(주소)이 쓰이는 시점에 준비되어 있는가?"** 언어의 실행 모델을 알면 이 부류의 버그가
한 가족으로 보이고, 어느 단계(컴파일? 링크? 실행?)에서 터졌는지로 원인을 좁힐 수 있습니다.
(강의 슬라이드의 `age++` 버그처럼 — **실행 모델을 모르면 못 잡는 버그**의 또 다른 예입니다.)

## CS 이해가 부족하면 생길 수밖에 없는 디버깅들

레거시 세대가 "당연히 확인하던 것들"이 요즘은 두꺼운 추상화(프레임워크·클라우드·AI)에 가려
보이지 않게 되면서, 세대 간 격차가 점점 벌어지고 있습니다. 공통 구조는 하나입니다 —
**증상은 위층에서 보이지만, 원인은 아래층에 있다.**

| 증상 | 순진한 해석 | 진짜 원인 후보 | 필요한 아래층 지식 |
|---|---|---|---|
| `OOM` | "램 부족 → 늘리자" | 누수 · 한도(`-Xmx`/limit) · 주소 공간 · 단편화 · 힙 오염 | 메모리·GC (01장 §5) |
| `nullptr`·`NameError`·`undefined` | "코드가 이상해요" | 초기화·링크·실행 **순서** | 번역·실행 모델 (01장 §3 · 위의 순서 절) |
| 한글이 `???` 로 깨짐 | "폰트 문제?" | **인코딩**(코드페이지 vs UTF-8) — C++판 `wmain` 사례 | 문자 표현 |
| `0.1 + 0.2 != 0.3` | "언어 버그?" | 부동소수점 **이진 표현** | 수 표현 |
| "어제는 됐는데 오늘 안 됨" | "컴퓨터 탓" | 의존성 **버전 범위**가 자동으로 새 버전을 당김 · 공유 런타임 자동 업데이트 | 고정 vs 범위 (02장) |
| "느려요" | "서버 늘리자" | GIL · GC pause · O(n²) 알고리즘 | 실행 모델·자료구조 |

도구가 좋아질수록 이 격차는 **더 벌어집니다** — AI 가 코드를 쓰는 시대(01장 §1)일수록, **아래층을
읽는 능력이 곧 디버깅 능력**입니다. 이 저장소의 목표가 바로 그 "아래층 읽기"의 첫걸음입니다.
