# 04. 기능별로 "전체 구현"을 나란히 — 왜 Python이 간단한가

> **이 문서의 핵심 원칙**: 코드를 한두 줄만 잘라(인라인) 비교하면 착시가 생긴다.
> 그 한 줄이 **동작하기 위해 뒤에 필요한 코드 전체**(클래스 선언, 인터페이스, 헤더/구현 분리,
> 소유권 관리 코드 등)를 함께 보여줘야 "어느 쪽이 정말 간단한가"가 드러난다. 그래서 이 문서는
> **기능 하나를 완성하는 데 필요한 코드를 통째로** 나란히 놓는다.

먼저 용어 몇 개(코딩 처음이라면):

- **보일러플레이트(boilerplate)**: 하는 일에 비해 "형식상 꼭 써야 하는" 반복 코드.
- **클래스(class)**: 데이터와 그 데이터를 다루는 함수를 묶은 설계도.
- **인터페이스 / 추상 클래스**: "이런 함수를 가져야 한다"는 **약속만** 정의한 껍데기.
  구현은 그 약속을 따르는 별도 클래스가 채운다. (부품 교체를 가능하게 하는 장치)
- **헤더(.hpp)/구현(.cpp) 분리**: C++ 는 "무엇이 있는지"(헤더)와 "어떻게 동작하는지"(구현)를
  **두 파일로 나눠** 쓰는 것이 관례다. 그래서 파일 수가 늘어난다.

---

## 왜 인라인 비교가 착시를 부르나 — 한 줄만 보면

판정식 한 줄만 떼어 보자.

```java
// Java
int diff = (player.code() - computer.code() + 3) % 3;
```
```python
# Python
diff = (player - computer + 3) % 3
```

거의 같아 보인다. 오히려 Java 가 별것 아닌 듯하다. 하지만 저 `player.code()` 가 되려면
**Java 는 enum 에 정수 필드와 접근자 메서드를 단 클래스 전체가 필요**하다. Python 은 `player`
에서 바로 빼는데, 그건 `Hand` 가 `IntEnum` 이라 **아무 추가 코드 없이** 되는 것이다.
그 "뒤에 숨은 전체"를 이제부터 통째로 펼쳐 본다.

## 1. 손(Hand): "정수 코드 + 이름 문자열 + 역조회"를 갖춘 열거형

목표 기능은 셋 다 같다 — 손을 0/1/2 정수로 다루고, 정수로 손을 되찾고(역조회), 이름
문자열("ROCK")도 얻는다.

### Python — 전체 구현 (5줄)
```python
class Hand(IntEnum):
    ROCK = 0
    PAPER = 1
    SCISSORS = 2
```
이게 **전부**다. `Hand(2)` 로 역조회, `Hand.ROCK.name` 으로 "ROCK", `Hand.ROCK - Hand.PAPER`
로 산술 — 셋 다 `IntEnum` 이 표준으로 준다.

### Java — 전체 구현 (Hand.java, 34줄)
```java
public enum Hand {
    ROCK(0),
    PAPER(1),
    SCISSORS(2);

    private final int code;          // 정수 코드를 담을 필드

    Hand(int code) {                 // 생성자에서 필드 채우기
        this.code = code;
    }

    public int code() {              // 코드 꺼내는 접근자
        return code;
    }

    public static Hand fromCode(int code) {   // 역조회를 직접 구현
        for (Hand hand : values()) {
            if (hand.code == code) {
                return hand;
            }
        }
        throw new IllegalArgumentException("유효하지 않은 손 코드: " + code);
    }
}
```

### C++ — 전체 구현 (hand.hpp, 41줄)
```cpp
enum class Hand : int { Rock = 0, Paper = 1, Scissors = 2 };

inline int to_code(Hand h) { return static_cast<int>(h); }

inline Hand hand_from_code(int code) {           // 역조회를 switch 로 직접
    switch (code) {
        case 0:  return Hand::Rock;
        case 1:  return Hand::Paper;
        case 2:  return Hand::Scissors;
        default: throw std::invalid_argument("유효하지 않은 손 코드: " + std::to_string(code));
    }
}

inline const char* to_string(Hand h) {           // 이름 문자열도 switch 로 직접
    switch (h) {
        case Hand::Rock:     return "ROCK";
        case Hand::Paper:    return "PAPER";
        case Hand::Scissors: return "SCISSORS";
    }
    return "UNKNOWN";
}
```

> **결론**: 정수코드·역조회·이름문자열이라는 같은 3가지 기능을 **Java 34줄 / C++ 41줄** 이
> 직접 만든다. Python 은 `IntEnum` 이 셋을 공짜로 줘서 **5줄**. 인라인 뺄셈 한 줄만 봤다면
> 절대 안 보였을 차이다.

## 2. 난수 주입(테스트 가능한 DI): 인라인이 가장 크게 속이는 사례

"컴퓨터 손을 보안 난수로 뽑되, **테스트에서는 난수를 원하는 값으로 바꿔 끼울 수 있게**" 하는
기능이다. 이걸 **의존성 주입(DI)** 이라 부른다.

### 인라인만 보면 (착시)
```python
self._rng = rng                                   # Python: 한 줄
```
```java
private final SecureRandom random = new SecureRandom();   // Java: 한 줄
```
둘 다 한 줄. 차이가 없어 보인다. 하지만 **"바꿔 끼울 수 있게"** 만들려면 이야기가 달라진다.

### C++ — 전체 구현 (교체 가능한 난수를 위해 필요한 코드 전부)

먼저 **약속(추상 클래스) + 보안 구현** 을 담은 파일 하나가 통째로 필요하다.
```cpp
// random_source.hpp — 31줄, 오직 "난수를 갈아끼울 수 있게" 하기 위한 파일
class RandomSource {                               // 약속(인터페이스)
public:
    virtual ~RandomSource() = default;
    virtual int next(int bound) = 0;               // "next(bound) 함수를 가져야 한다"
};

class SecureRandomSource : public RandomSource {   // 그 약속을 지키는 보안 구현
public:
    int next(int bound) override {
        std::uniform_int_distribution<int> dist(0, bound - 1);
        return dist(rd_);
    }
private:
    std::random_device rd_;
};
```
그리고 게임 클래스는 이 난수원을 **소유(unique_ptr)** 하고 생성자로 주입받도록, 멤버 선언과
생성자 코드를 더 써야 한다.
```cpp
// rock_paper_scissors.hpp (발췌) — 소유권 관리 부분
class RockPaperScissors {
public:
    explicit RockPaperScissors(
        std::unique_ptr<RandomSource> rng = std::make_unique<SecureRandomSource>());
private:
    std::unique_ptr<RandomSource> rng_;            // 누가 메모리를 정리할지까지 관리
};
```
```cpp
// rock_paper_scissors.cpp (발췌) — 생성자 구현
RockPaperScissors::RockPaperScissors(std::unique_ptr<RandomSource> rng)
    : rng_(std::move(rng)) {}
```
테스트에서 교체하려면 **또 다른 구현 클래스**(예: 항상 2를 주는 `FixedRandomSource`)를 만들어
주입한다. → 이 기능 하나에 **약 40줄 + 파일 여러 개**.

### Python — 전체 구현 (1줄)
```python
def __init__(self, rng: Callable[[int], int] = secrets.randbelow) -> None:
    self._rng = rng
```
함수가 **일급 객체**(값처럼 주고받을 수 있음)라, "난수원"이 그냥 함수 하나면 된다. 별도 인터페이스도,
구현 클래스도, 소유권 관리도 없다. 테스트 교체도 한 줄:
```python
game = RockPaperScissors(rng=lambda _bound: 2)     # 컴퓨터는 항상 SCISSORS(2)
```

> **결론**: 같은 "교체 가능한 보안 난수"를 위해 **C++ 는 전용 파일 + 추상클래스 + 소유권 관리 코드 ~40줄**,
> Python 은 **1줄**. 인라인(`self._rng = rng` 한 줄)만 봤다면 Python 이 오히려 단순하다는 게
> 안 보였을 것이다 — 뒤에 붙는 스캐폴딩이 없다는 게 핵심이니까.

## 3. 다국어 파서: 정규화 함수와 헤더/구현 분리

"공백 제거 + 소문자화 후 별칭표에서 손을 찾는다." 별칭 데이터(19개)는 셋 다 비슷하지만,
**그 데이터를 감싸는 코드**의 양이 다르다.

### Python — 전체 구현 (약 12줄 + 별칭 데이터)
```python
def parse_hand(text: str) -> Hand:
    if text is None:
        raise ValueError("입력이 없습니다.")
    hand = _ALIASES.get(text.strip().lower())      # 정규화가 이 한 줄
    if hand is None:
        raise ValueError(f"알 수 없는 손 입력: {text!r}")
    return hand
```

### C++ — 전체 구현 (hand_parser.hpp 15줄 + hand_parser.cpp 71줄, 두 파일)
C++ 는 "정규화(공백 제거 + 소문자화)"를 **직접 함수로** 만들어야 한다.
```cpp
// hand_parser.cpp (발췌) — Python 의 .strip().lower() 한 줄에 해당하는 부분
std::string normalize(const std::string& in) {
    const char* ws = " \t\r\n";
    std::size_t b = in.find_first_not_of(ws);
    if (b == std::string::npos) return "";
    std::size_t e = in.find_last_not_of(ws);
    std::string s = in.substr(b, e - b + 1);
    std::transform(s.begin(), s.end(), s.begin(),
                   [](unsigned char c) { return static_cast<char>(std::tolower(c)); });
    return s;
}
```
여기에 별칭표·조회·예외 처리(59~68행)와, 함수 **선언**만 담은 별도 헤더 파일(`hand_parser.hpp`)까지
합쳐 **두 파일 86줄**.

### Java — 전체 구현 (HandParser.java, 57줄)
`Map.ofEntries(Map.entry(...))` 로 별칭표를 만들고, `private` 생성자, `null` 검사,
`strip().toLowerCase(Locale.ROOT)` 정규화를 갖춘 **클래스 한 개**.

> **결론**: 별칭 데이터는 비슷해도, Python 은 (1) 헤더/구현 분리가 **없고** (2) 정규화가
> `.strip().lower()` **한 줄**이다. C++ 는 정규화 함수만 12줄을 손수 짠다.

## 4. 불변 결과 타입: Python 이 "항상" 이기는 건 아니다 (정직한 비교)

과장하지 않기 위해 반대 사례도 본다. "값 3개를 묶은 불변 데이터"는 —

```java
public record GameResult(Hand player, Hand computer, Outcome outcome) {}   // Java: 1줄!
```
```python
@dataclass(frozen=True)
class GameResult:                # Python: 4줄
    player: Hand
    computer: Hand
    outcome: Outcome
```
```cpp
struct GameResult {              // C++: 헤더 파일 하나(include/namespace 포함 15줄)
    Hand player;
    Hand computer;
    Outcome outcome;
};
```

여기선 **Java `record` 가 1줄로 가장 짧다**. Python 의 이점은 "모든 것"이 아니라, **클래스/인터페이스/
헤더 스캐폴딩을 새로 만들어야 하는 곳**에 집중된다는 걸 정직하게 알아 두자.

## 5. 웹 계층: 3개 클래스 vs 함수 1개

"GET 요청을 받아 게임하고, 잘못된 입력이면 400" 을 만드는 데:

- **Java (전체)**: `GameController`(30줄) + `ApiExceptionHandler`(18줄) + 부트 진입점
  `RockPaperScissorsApplication`(23줄) = **3개 클래스, 71줄**.
- **C++**: 웹 계층 없음(범위 제외).
- **Python (전체)**: 앱 객체 + 함수 하나.
```python
app = FastAPI()
_game = RockPaperScissors()

@app.get("/api/play")
def play(h: str) -> dict[str, str]:
    try:
        r = _game.play(h)
    except ValueError as exc:
        raise HTTPException(status_code=400, detail=str(exc)) from exc
    return {"player": r.player.name, "computer": r.computer.name, "outcome": r.outcome.name}
```

## 요약 — 같은 기능, 필요한 "전체 구현" 규모

| 기능 | Java (전체) | C++ (전체) | Python (전체) |
|---|---|---|---|
| 손 열거형 (코드+역조회+이름) | 34줄 (1파일) | 41줄 (1헤더) | **~5줄** |
| 테스트 가능한 난수 주입 | 인터페이스 도입 필요 | ~40줄 (전용 파일+소유권 관리) | **1줄** |
| 다국어 파서 (정규화 포함) | 57줄 (1파일) | 86줄 (헤더+구현 2파일) | **~12줄** |
| 불변 결과 타입 | **1줄 (record)** ← Java 승 | 15줄 (1헤더) | 4줄 |
| 웹 엔드포인트 + 에러 | 71줄 (3개 클래스) | 없음 | **~10줄** |

## 왜 이런 차이가 나나 (원리)

- **Java/C++ 는 "구조를 직접 세운다"**: 정수 코드·역조회·이름·교체 지점을 **클래스/인터페이스로
  손수** 만들어야 하고, C++ 는 헤더/구현 분리와 소유권(`unique_ptr`)까지 문법이 요구한다.
- **Python 은 "구조를 언어가 준다"**: `IntEnum`·`@dataclass`·일급 함수·덕 타이핑이 그 스캐폴딩을
  **표준으로 대신** 제공한다. 그래서 "새 클래스/파일을 만들어야 할 일"이 몇 줄로 줄어든다.
- **단, 만능은 아니다**: Java `record` 처럼 이미 간결한 장치가 있는 기능에선 차이가 작거나 역전된다.
  Python 의 이점은 **"보통은 클래스·인터페이스·파일을 새로 세워야 하는 일을 몇 줄로 끝낼 수 있는 곳"**
  에서 극대화된다 — 이 저장소에선 난수 주입·열거형·웹 계층이 대표적이다.

## 6. 규모가 커지면 격차는 더 벌어진다 — Spring MVC · OOP · GC

이 저장소는 작아서 Spring 의 진짜 무게가 거의 안 드러난다. 실제 서비스처럼 커지면
Java 쪽에는 아래 구조들이 얹히면서 코드·설정이 훨씬 빠르게 늘어난다.

- **Spring MVC 계층화**: 요청 하나를 처리하는 데도 `Controller`(입출력) → `Service`(로직) →
  `Repository`(DB) 로 계층을 나누고, 그 사이를 **DI 컨테이너**가 `@Autowired`/`@Bean` 으로 연결한다.
  여기에 어노테이션 수십 개, **Bean 생명주기**, 설정 파일(`application.yml`), AOP(공통 관심사),
  트랜잭션 관리 등이 더해진다. → "요청 → 응답" 한 흐름에 **여러 파일·여러 규약**이 붙는다.
- **OOP 세리머니**: 유연성·확장성을 위해 인터페이스와 구현을 분리하고(2절의 난수 주입이 그 축소판),
  추상 클래스, 디자인 패턴(팩토리·전략·템플릿 메서드), 제네릭을 도입한다. **클래스 수가 늘어난다.**
- **GC / 메모리**: JVM 은 규모가 커지면 **GC 튜닝**(힙 크기, GC 알고리즘 선택, 지연 시간 관리)이
  실무 관심사가 된다. C++ 는 아예 **소유권 설계**(누가 만들고 누가 지우나)를 직접 해야 한다.
  Python 도 GC 가 있지만 대개 신경을 덜 쓴다.

같은 확장을 Python 으로 하면 대체로 더 가볍다 — FastAPI 는 의존성 주입을 **함수 인자**로,
데이터 검증을 **Pydantic 모델**로, 라우팅 분리를 **APIRouter** 로 처리해 계층이 완만하게 는다.

### 그런데 — "복잡함 = 나쁨"이 아니다 (정직한 균형)

Spring/C++ 의 그 복잡성 상당수는 **낭비가 아니라 투자**다.

- **대규모 협업**: 계층·인터페이스·타입은 수십 명이 수십만 줄을 함께 고칠 때 **안전망**이 된다.
  컴파일러가 실수를 미리 잡아 주고, 구조가 "어디를 고쳐야 하는지"를 강제한다.
- **성능·안정성**: JIT·GC 튜닝·정적 타입은 대량 트래픽에서 유리하다.
- Python 의 가벼움은 **소규모·빠른 반복**에 최적이지만, 대규모에서는 정적 타입 부재·런타임 오류·
  실행 속도가 약점이 되어 **타입 힌트·테스트·계층화를 스스로 도입**해 그 격차를 메워야 한다.

> **결론**: "복잡함 vs 간단함"이 아니라 **문제 규모에 맞는 도구가 다르다**. 가위바위보 정도에선
> Python 이 압도적으로 편하고(이 문서의 사례들), 대규모 엔터프라이즈에선 Spring 의 구조가 값을 한다.
> 이 저장소의 목표는 "Python 이 항상 옳다"가 아니라, **작은 문제에서 언어가 요구하는 스캐폴딩의 무게
> 차이를 눈으로 보는 것**이다.
